# AI 代码审核标准

当 AI 代码在日常工作中占比超过 50% 时，没有一套审核标准就是在裸奔。人类写的 bug 和 AI 写的 bug 不是一类东西，审核标准需要对症。

## AI 代码的特有风险

| AI 特有风险               | 表现                                                              | 严重程度 |
| ------------------------- | ----------------------------------------------------------------- | -------- |
| **幻觉（Hallucination）** | 编造不存在的 API、函数、库、命令行参数                            | P0       |
| **版本过时**              | 用了已弃用的语法/API（训练数据截止日期之前的版本）                | P0       |
| **项目约定漂移**          | 不按 CLAUDE.md 或现有代码风格写，引入新模式                       | P0       |
| **过度工程**              | 为一个简单任务加了三层抽象、工厂模式、策略模式                    | P1       |
| **防御性噪音**            | 无意义的 try/except、对不可能发生的情况做校验                     | P1       |
| **浅表修复**              | 绕过错误而非修复根因（如 `try: except: pass`）                    | P1       |
| **测试空壳**              | 测试调用了函数但 `assert True`、覆盖率达标但没 assert 实际行为    | P1       |
| **上下文失忆**            | 多文件改动时前后不一致，或改了 A 忘了同步改 B                     | P1       |
| **安全漏洞**              | SQL 拼接、硬编码密钥、不校验用户输入                              | P0       |
| **许可证污染**            | 生成了逐字复制的受限源码（少见但致命）                            | P0       |

### 幻觉：最隐蔽的风险

AI 幻觉是最难发现的，因为代码看起来特别合理：

- 人类写错 API 通常是"写成了另一个真实存在的 API"，搜一下就能发现。
- AI 幻觉的 API 往往**完美符合命名直觉**（比如 `database.rollback_to_savepoint("name")` 听起来就该存在），但翻官方文档才发现没这东西。

**防御手段**：AI 写的任何库/API 调用，必须对照官方文档或 "go to definition" 跳转验证，不能基于"这名字对，应该行"。

### 案例：审核者自己也会幻觉（非代码场景）

幻觉不仅出现在代码里，**任何"归纳 + 推断"类的任务**都可能发生——包括 AI 自己写的审查、对比、文档。

**真实案例**：在撰写 Claude Code 三种 review 工具对比笔记（`cc/review-skills-comparison.md`）时，AI 把内置 `/review` skill 定位为"审查本地代码的轻量工具"。

**实际情况**：`/review` 在 Claude Code 命令列表里的官方描述明确是 **"Review a pull request"**——它是针对 PR 的，不是本地代码。

**幻觉是怎么发生的**：

1. AI 观察到另外两个工具（`/code-review` 和 `/pr-review-toolkit`）都强调 PR/GitHub 集成
2. 推测"总得有一个轻量、本地的选项吧"（**叙事对称性偏好**）
3. 把 `/review` 脑补成这个"本地"选项，**没有验证其官方描述**
4. 进一步在决策树里推荐"本地代码用 /review"，**错误级联放大**

### 这个案例的教训

- **审核者不免疫幻觉**——做审核、对比、文档这类任务时，AI 同样会脑补
- **叙事完整性是幻觉的高发区**——"三个工具，总得有一个是轻量的"这种对称归纳，最容易产生似是而非的结论
- **可验证事实 > 合理推测**——官方描述、截图、文档应优先于"听起来对"的判断
- **幻觉会级联**：错误的定位 → 错误的对比 → 错误的决策树 → 误导读者

### 防御手段（延伸到审核与文档工作）

1. 写对比类内容时，**先把每个对象的官方描述逐字引用**，不要凭印象归纳
2. 审核 AI 输出时，**警惕"推测性论断"**——凡是没有直接证据的判断，都要追问依据
3. **让 AI 自己标注"我在推测"vs"我在引用"**，分离可信度等级
4. 用户/审查者的反问是最珍贵的校正信号——截图、原始描述、反驳证据比任何自审机制都有效

## 分层审核清单

按重要性分三层，顺序执行：

### P0：必查（每个 AI 生成的 PR 都要过）

1. **幻觉检查**：新引入的 API、库函数、CLI 参数，跳转定义或查官方文档确认存在。
2. **安全扫描**：搜索 `password`、`secret`、`api_key`、`token`、`eval(`、`exec(`、拼接 SQL。
3. **项目约定**：与 CLAUDE.md 对照，检查命名、目录、错误处理约定是否符合。
4. **依赖版本**：新增的 import 是否已在 `package.json` / `requirements.txt` 里、版本是否现行。

### P1：重要（代码进主干前必审）

5. **过度工程**：问自己"这个抽象解决的问题真的存在吗？" 三个相似的行往往比一个抽象更好。
6. **错误处理**：捕获的异常有没有具体处理逻辑？还是只是 `except Exception: pass`？
7. **测试有效性**：测试失败时，能精确定位 bug 吗？还是只会报 "AssertionError"？
8. **跨文件一致性**：改了 interface 有没有同步改所有 implementation？

### P2：优化（审美层面，不 block 合并）

9. **命名语义**：变量名是否清晰、不冗余。
10. **注释必要性**：注释说明的是 *WHY* 还是 *WHAT*？WHAT 的注释应删（代码本身就是 WHAT）。
11. **性能热点**：循环里有没有做 O(n²) 的事、能不能批量化。

## 审核流程：边写边审 > 最终审

**不要等 AI 写完一大段再审**——这是最常见的错误。

| 时机           | 做什么                                                   |
| -------------- | -------------------------------------------------------- |
| **Prompt 前**  | 想清楚约束条件，写进 prompt（而不是事后发现不符合再改） |
| **Diff 中**    | AI 每完成一小块，快速扫 diff（看是否跑偏）              |
| **Commit 前**  | 过一遍 P0 清单 + 本地运行一次（跑测试、启动 dev server）|
| **PR 中**      | 让另一个 AI（或 code-review agent）独立审查一次         |
| **合并后**     | CI 通过 ≠ 代码正确，关注线上监控（Sentry）一周           |

## 核心原则

### 信任但验证

- **信任**：AI 的大部分输出是对的，不要每行都人工复写。
- **验证**：AI 的**跳跃点**（新 API、新文件、新抽象）必须验证，因为这是幻觉发生概率最高的地方。

### 边际审核成本应匹配边际失败代价

不同任务的审核强度应该不一样：

| 任务类型             | 审核强度                                         |
| -------------------- | ------------------------------------------------ |
| 改文案、调样式       | 轻审（看 diff 是否合理即可）                     |
| 改业务逻辑           | 中审（跑测试 + 人工读一遍）                      |
| 改安全/支付/数据迁移 | 重审（多 AI 审核 + 人工逐行 + staging 验证）     |

**不要对所有 AI 代码一视同仁**——那样要么太慢，要么在安全关键路径放水。

## 工具支持

- **Claude Code 内置**：`feature-dev:code-reviewer` agent、`/security-review` skill
- **静态检查**：pyright / tsc / eslint 能捕获大部分幻觉（不存在的符号）
- **测试**：跑测试比读代码更能发现浅表修复
- **Diff 可视化**：`git diff --word-diff` 看局部改动
- **多 AI 交叉审核**：让另一个模型（或同模型不同 session）独立审一遍，捕获单次推理的盲点
