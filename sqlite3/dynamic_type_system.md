SQLite 的数据类型系统非常独特，它不像 MySQL 或 PostgreSQL 那样有严格的类型限制，而是采用了 **动态类型系统 (Dynamic Type System)**，这意味着数据的类型由**值本身**决定，而不是由存储它的列（容器）决定。

要全面理解 SQLite 的数据类型，需要区分两个核心概念：**存储类别 (Storage Classes)** 和 **类型亲和性 (Type Affinities)**。

---

### 🧬 核心一：存储类别 (Storage Classes)

存储类别描述的是 **值本身** 的类型，所有存入 SQLite 数据库的值，都会被归为以下 5 种之一。

| 存储类别    | 说明                                                                                              |
| :---------- | :------------------------------------------------------------------------------------------------ |
| **NULL**    | 空值。表示缺失的数据，与数字 `0` 或空字符串 `''` 完全不同。                                       |
| **INTEGER** | 带符号的整数。SQLite 会根据数值的大小，智能地使用 1, 2, 3, 4, 6 或 8 个字节来存储，非常节省空间。 |
| **REAL**    | 8 字节的 IEEE 浮点数。用于存储小数值。                                                            |
| **TEXT**    | 文本字符串。默认使用 UTF-8、UTF-16BE 或 UTF-16LE 编码存储。                                       |
| **BLOB**    | 二进制大对象。数据完全按照输入的原始形式存储，不做任何转换。适合存储图片、文件等。                |

---

### 📌 核心二：类型亲和性 (Type Affinities)

类型亲和性是 SQLite 最特别的概念。当你在 `CREATE TABLE` 语句中指定某列的类型（例如 `VARCHAR(255)`）时，SQLite 并不会强制该列只能存储字符串，而是为它分配一个 **亲和性 (Affinity)**。这个亲和性更像是一个 **推荐类型**，SQLite 会**尝试**将存入的数据转换为列的亲和性类型，但绝不会因类型不匹配而报错。

类型亲和性分为以下 5 种：

| 亲和性      | 说明                                                                                                                 |
| :---------- | :------------------------------------------------------------------------------------------------------------------- |
| **TEXT**    | 存储数据时，会优先尝试将数据转换为 `TEXT` 类型。例如，存入整数 `123` 会被存为字符串 `'123'`。                        |
| **NUMERIC** | 最灵活的亲和性。如果存入的文本能无歧义地转换为数字（如 `'123.45'`），则存为 `INTEGER` 或 `REAL`；否则，存为 `TEXT`。 |
| **INTEGER** | 行为与 `NUMERIC` 相似，但倾向于将数据转换为 `INTEGER` 类型。                                                         |
| **REAL**    | 类似于 `NUMERIC`，但会强制将整数值转换为浮点数存储。                                                                 |
| **BLOB**    | 不尝试对数据进行任何转换，存入什么就是什么，所以也叫 `NONE` 亲和性。                                                 |

**SQL 列类型与亲和性映射**

下面列出了我们常用的 SQL 列类型，以及它们在 SQLite 中被分配了什么亲和性：

| 亲和性      | 对应的常见列类型声明                                                           |
| :---------- | :----------------------------------------------------------------------------- |
| **INTEGER** | `INT`, `INTEGER`, `TINYINT`, `SMALLINT`, `MEDIUMINT`, `BIGINT`, `INT2`, `INT8` |
| **TEXT**    | `CHARACTER(20)`, `VARCHAR(255)`, `TEXT`, `CLOB`                                |
| **BLOB**    | `BLOB`                                                                         |
| **REAL**    | `REAL`, `DOUBLE`, `FLOAT`                                                      |
| **NUMERIC** | `NUMERIC`, `DECIMAL(10,5)`, `BOOLEAN`, `DATE`, `DATETIME`                      |

### 🗓️ 日期与时间

SQLite **没有**专门用于存储日期和时间的 `DATE` 或 `DATETIME` 存储类别。通常有以下几种存储策略：

| 方法            | 说明                                                                                          |
| :-------------- | :-------------------------------------------------------------------------------------------- |
| **TEXT (推荐)** | 使用 ISO8601 格式的字符串（如 `'2026-04-23 10:30:00'`）。这种格式可读性高，也支持排序和比较。 |
| **INTEGER**     | 存储 Unix 时间戳（自 1970-01-01 以来的秒数）。这种方式存储空间小，计算效率高。                |
| **REAL**        | 存储儒略日（Julian day numbers）数值。这是一种天文纪年法，在日常开发中较少使用。              |

SQLite 提供了丰富的内置 [日期和时间函数](https://www.sqlite.org/lang_datefunc.html) 来处理以上三种格式的数据。

---

### ✅ 布尔值

SQLite 同样**没有**原生的 `BOOLEAN` 存储类别。它使用整数 `0` 和 `1` 来表示。

- **存储**：`FALSE` 存为 `0`，`TRUE` 存为 `1`。
- **关键字**：从 3.23.0 版本开始，`TRUE` 和 `FALSE` 可以作为关键字使用，但它们本质上只是 `1` 和 `0` 的同义词。
- **类型声明**：你依然可以声明列为 `BOOLEAN` 类型，它会自动获得 `NUMERIC` 亲和性，但这并不强制该列只能存 0 和 1。

---

### 🔒 最佳实践：现代 STRICT 表

如果你更倾向于传统关系型数据库那种严格的类型检查，SQLite 从 3.37.0 版本开始提供了 **`STRICT` 表**。创建表时在末尾加上 `STRICT` 关键字，可以强制列只能存储其声明的类型。

```sql
-- 传统表：灵活，可以存任何类型
CREATE TABLE users (
    id INTEGER PRIMARY KEY,
    name TEXT
);

-- STRICT 表：严格类型检查，name 列只能存 TEXT, BLOB 或 NULL
CREATE TABLE users (
    id INTEGER PRIMARY KEY,
    name TEXT
) STRICT;
```

### 💎 总结：如何选择？

为了方便你快速决策，下面这个表格总结了常见场景下的推荐类型：

| 场景                 | 推荐使用的列类型 | 备注                                                                |
| :------------------- | :--------------- | :------------------------------------------------------------------ |
| **主键 ID**          | `INTEGER`        | 若与 `PRIMARY KEY` 联用，该列会成为 `rowid` 的别名，实现自增。      |
| **普通整数**         | `INTEGER`        | 最常见的选择。                                                      |
| **浮点数**           | `REAL`           | 对于金融等需要高精度的数据，建议用 `INTEGER` 存储最小单位（如分）。 |
| **字符串**           | `TEXT`           | 用于姓名、标题、描述等。                                            |
| **布尔值**           | `INTEGER`        | 用 `0` (false) 和 `1` (true) 表示。                                 |
| **日期/时间**        | `TEXT`           | 使用 ISO8601 格式 (`YYYY-MM-DD HH:MM:SS`)，可读性最好。             |
| **日期/时间 (计算)** | `INTEGER`        | 使用 Unix 时间戳，计算效率高。                                      |
| **二进制数据**       | `BLOB`           | 用于存储文件、图片、序列化对象等。                                  |
| **需要严格类型检查** | `...` + `STRICT` | 在 `CREATE TABLE` 语句末尾加上 `STRICT` 关键字。                    |
