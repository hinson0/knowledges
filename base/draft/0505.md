你这个总结**方向是对的,但不够精确**。让我帮你把这个理解再校准一下,因为这个区分会直接影响你以后做 LLM 评估时的指标选择。

## 你的总结里对的部分

"除非本身是数值,否则不能算方差"——这部分完全正确。这是统计学最基础的类型规则,记住这一条就能避开 90% 的踩坑。所以你以后看到一个数据,第一步永远是问"这是数值还是类别",再决定用什么工具。

## 但"求众数就可以了"这一步,我得给你拨一下

众数(以及众数占比)只是**类别数据分散度指标里最简单的那一个**,它够用,但不是最全面的。我用两个具体例子让你看为什么。

假设你有两组 LLM 输出的 sentiment 结果,各 10 次:

A 组:`[neg, neg, neg, neg, neg, neg, neg, neg, pos, pos]`
B 组:`[neg, neg, neg, neg, neg, neg, neg, neg, pos, neutral]`

**两组的众数都是 negative,众数占比都是 80%**。如果你只看众数,会得出"两组一样稳"的结论。但实际上 B 组比 A 组更不稳——因为 B 组里"非主流答案"分散在两个不同类别里(说明模型摇摆得更厉害),A 组的"非主流答案"都集中在一个类别里(说明模型至少在"非 negative 时统一选 pos"这一点上有一致性)。

这种细微差异**只有基尼或熵能捕捉到**:A 组基尼 = 1 - 0.8² - 0.2² = 0.32,B 组基尼 = 1 - 0.8² - 0.1² - 0.1² = 0.34。差异很小但方向是对的——B 组确实更分散。

再看一个更极端的例子:

C 组:`[neg, neg, neg, neg, neg, pos, pos, pos, pos, pos]`(众数占比 50%)
D 组:`[neg, neg, neg, neg, pos, pos, pos, neutral, neutral, neutral]`(众数占比 40%)

按众数占比看,C 组(50%)比 D 组(40%)更集中。但按熵看,C 组熵 = 1 bit,D 组熵 ≈ 1.57 bit——D 组确实更分散。这次众数和熵给出的判断方向是一致的,但**衡量"分散到什么程度"的精细度,熵远远高于众数占比**。

## 真正实用的判断框架

我给你一个工程上够用的简化版决策树,你以后选指标按这个走就行。

**如果你只关心"模型在最常见的判断上有多大概率正确"**——这是产品视角,因为用户主要看的是模型的"一般表现"——那么**众数占比就够了**。比如老板问你"我们的 LLM 在情感分类任务上表现稳不稳",你回答"80% 的输出落在主流判断上",这个数字直观、好理解、能用来谈业务。

**如果你要做严谨的 prompt 评估、对比不同方案、写技术报告**——这时候光看众数占比会让你错过细节,需要**基尼或熵**来捕捉"次要类别的分布形态"。两个之中我更推荐**基尼**,因为它公式简单、值在 0-1 之间、不用理解信息论也能直觉理解(就是"随机抽两个抽到不同类别的概率")。

**如果你做的是机器学习/信息论相关任务**——比如训练分类器、做特征选择、计算 KL 散度——那必须用**熵**,因为熵和这些领域的其他工具(交叉熵损失、信息增益、互信息)是数学上一脉相承的,换成基尼会割裂。

## 一个值得记住的工程心法

我想让你记住一句话:**"指标越多,你就越知道自己在测什么"**。新手做评估容易陷入"找一个完美指标"的误区,以为有个银弹能把"模型好不好"压缩成一个数。真正成熟的工程师会做的是**算 3-5 个互补指标,看它们之间是否一致**——如果一致,结论可信;如果矛盾(比如众数占比变高了但基尼也变高了),那说明这个改动有副作用,需要深挖。

回到你最初要做的那个三组对比实验(纯 Pydantic、纯 few-shot、Pydantic + few-shot),我建议这样组合指标:对 sentiment 字段算**众数占比 + 基尼**(类别数据),对 confidence 字段算**均值 + 标准差**(数值数据),对 reason 字段算**长度均值 + 长度变异系数 CV**(数值数据,用 CV 是因为不同组平均长度可能不同),对原始字符串算**多行格式占比**(类别数据,只有"单行/多行"两个值,直接看比例就够了)。这样一组实验会给你 7 个指标,信息密度足够你看出哪些维度被改善、哪些没被改善。

## 修正后的总结应该是什么

你那句话改成这样会更准确:**"如果数据是数值,用方差/标准差;如果数据是字符串类别,用众数占比就能解决大部分问题,严谨场合再加上基尼或熵。"**

这其中有个隐藏的、但非常实用的工程小细节我再单独提一下——**有时候你拿到的"数值"其实不是真数值**。比如 LLM 输出的 confidence 是 0-1 之间的数,看起来是数值,但模型其实只爱吐出 0.7、0.75、0.8、0.85 这几个"档位"(因为训练数据里这些值最常见)。这种情况下 confidence 表面是数值,本质上是**伪数值的离散类别**,你算它的方差能算出来,但要警惕"几个档位之间的间隔均匀吗"这种隐藏假设。这是 LLM 评估里一个隐蔽的坑,跑实验时多看几次原始输出能帮你发现。

你跑你那三组实验时可以观察一下 confidence 字段——大概率你会看到清一色的 0.7/0.75/0.8/0.85,几乎不会出现 0.73 或 0.82 这种数。这就是这个"伪数值"现象的具体表现。LLM 不是真的在做精细的概率估计,它只是在"这种场合下我应该说哪个常见的数"里挑一个。这个观察对理解 LLM 输出的本质非常重要——**模型吐出的每个数字都是一个 token,它只会用训练数据里高频出现的 token,而不会真的做数学计算**。

## 直接结论先放在前面

你这个数组**不能直接用 `statistics.variance()` 这种函数算"方差"**,会报错。但你**完全可以衡量它的"分散程度"**——只是要换一套指标。这两件事听起来矛盾,实际上揭示了统计学里一个非常重要的区分:**数据类型决定了你能用什么统计工具**。

## 为什么不能直接算方差

要回答这个问题,你得先理解"方差"这个词在数学上的精确定义。方差的公式是这样的:**先算所有数值的平均值,然后算每个数值与平均值的差的平方,再求平均**。这个公式里有两个隐含假设——第一,数据必须是**数值型**(能做减法);第二,数据点之间的"距离"必须有意义(0.7 和 0.8 的距离是 0.1,这个 0.1 是有物理意义的)。

现在看你的数组 `['negative', 'negative', 'negative', 'negative', 'positive', 'neutral']`。这是**字符串**,你不能算 `negative - positive` 等于多少,这个减法在数学上没有定义。你可能会想——那我把它们映射成数字呢?比如 negative=0, neutral=1, positive=2,然后算这串数字的方差?**这恰恰是个陷阱,我专门讲一下,因为你以后一定会遇到有人这么干**。

## 那个陷阱:为什么不能"映射成数字再算方差"

假设我们映射成 negative=0, neutral=1, positive=2。那么你的数组变成 `[0, 0, 0, 0, 2, 1]`,平均值是 0.5,方差是 0.7。看起来一切都很顺利对不对?但问题来了——**这个 0.7 没有任何意义**。

为什么没有意义?因为这个映射隐含了一个错误假设:**negative 到 neutral 的"距离"等于 neutral 到 positive 的"距离",且都等于 1**。但事实上情感这三个标签之间根本没有"距离"概念,它们是**相互独立的类别**,不存在"哪两个更接近"。如果我换一种映射 negative=-5, neutral=0, positive=5,数组就变成 `[-5,-5,-5,-5,5,0]`,方差变成 17.5。同一份数据,换个映射方差就变了——这说明这个数字本身没有真实含义,只是你映射方式的副产物。

这种数据类型,统计学里叫**类别数据(categorical data)**或者**名义变量(nominal variable)**,与之相对的是**数值数据(numerical data)**,比如温度、身高、confidence。**类别数据有自己专门的"分散程度"指标,不能套用方差**。这是统计学第一节课就会强调的事情,但很多工程师没系统学过统计,就容易踩坑。

## 类别数据正确的"方差"指标

那对你这个数组,**正确的"分散程度"该怎么衡量**?有三个工业界常用的指标,我从直觉到严谨给你讲。

**指标一:众数占比(Mode Proportion)**。这是最直觉的——出现最多的那个类别占了多大比例?你的数组里 negative 出现 4 次,占 4/6 ≈ 66.7%。这个数越接近 100% 说明越集中,越接近 1/类别数(三类的话就是 33%)说明越分散。这个指标我们上一轮的 `analyze_variance` 函数里已经用过了,叫 `sentiment_consistency`。

**指标二:基尼不纯度(Gini Impurity)**。这个名字听起来高深,其实公式很简单——`1 - Σpᵢ²`,其中 pᵢ 是每个类别的出现比例。你的数组里 negative 占 4/6,positive 占 1/6,neutral 占 1/6,所以基尼不纯度 = 1 - (4/6)² - (1/6)² - (1/6)² = 1 - 0.444 - 0.028 - 0.028 = **0.5**。这个值在 0 到 (1 - 1/类别数) 之间,0 表示完全集中(全部一样),最大值表示完全均匀分布。**机器学习里的决策树用的就是这个指标来选分裂特征**,所以你以后做 ML 一定会再见到它。

**指标三:熵(Entropy)**。公式是 `-Σpᵢ × log₂(pᵢ)`,单位是 bit。你的数组熵 = -(4/6)·log₂(4/6) - (1/6)·log₂(1/6) - (1/6)·log₂(1/6) ≈ 0.39 + 0.43 + 0.43 = **1.25 bit**。完全集中时熵=0,完全均匀分布在 3 个类别时熵=log₂(3)≈1.58。熵的好处是它有明确的信息论意义——表示"猜出下一次结果平均需要问多少次是非题"。**这是信息论的核心指标**,在 NLP、推荐系统、机器学习里到处都是它的影子。

这三个指标是**正相关的**——某个数据集众数占比高,基尼也低,熵也低。它们衡量的是同一件事(类别分散程度),只是从不同角度切入。在工程实战中你随便选一个用就行,我个人偏好基尼不纯度,因为它公式简单、值在 0-1 范围、容易跨实验对比。

## 给你一段能直接跑的代码

下面这段代码把这三个指标都算出来了,你可以直接用:

```python
import math
from collections import Counter


def categorical_dispersion(labels: list[str], all_categories: list[str] = None) -> dict:
    """
    计算类别数据的分散程度指标。

    参数 labels:类别字符串数组,比如 ['negative', 'negative', 'positive']
    参数 all_categories:所有可能的类别(可选)。传这个的好处是即使你的样本里
                        某个类别一次都没出现,计算"理论最大熵"时也能正确归一化。

    返回三个核心指标 + 一些辅助信息。
    """
    n = len(labels)
    if n == 0:
        return {"error": "空数组"}

    # 统计每个类别出现的次数
    counter = Counter(labels)

    # ---------- 指标一:众数占比 ----------
    # most_common(1) 返回 [(出现最多的类别, 次数)],我们取 [0][1] 拿到次数
    mode_count = counter.most_common(1)[0][1]
    mode_proportion = mode_count / n

    # ---------- 指标二:基尼不纯度 ----------
    # 公式:1 - Σ(pᵢ²)
    # 直觉理解:从数组里随机抽两次,两次抽到不同类别的概率
    # 全部相同时 = 0(永远抽到一样的);完全均匀时趋近于 1
    gini = 1 - sum((count / n) ** 2 for count in counter.values())

    # ---------- 指标三:香农熵(以 2 为底,单位 bit)----------
    # 公式:-Σ(pᵢ × log₂(pᵢ))
    # 注意 pᵢ=0 时 log 没定义,但 0·log(0) 数学上规定为 0,所以直接跳过
    entropy = 0.0
    for count in counter.values():
        if count > 0:  # 跳过 0,避免 log(0) 错误
            p = count / n
            entropy -= p * math.log2(p)

    # 如果用户告诉我们"理论上有几个类别",可以算"归一化熵"(0-1 之间)
    # 这个在跨实验对比时很有用 —— 不同实验可能样本里出现的类别数不同,
    # 直接比熵不公平,归一化后就能比
    normalized_entropy = None
    if all_categories:
        max_entropy = math.log2(len(all_categories))
        if max_entropy > 0:
            normalized_entropy = entropy / max_entropy

    return {
        "n": n,
        "distribution": dict(counter),         # 每个类别的具体次数
        "mode": counter.most_common(1)[0][0],  # 最常见的类别本身
        "mode_proportion": mode_proportion,    # 众数占比,越大越集中
        "gini_impurity": gini,                  # 基尼不纯度,越小越集中
        "entropy_bits": entropy,                # 香农熵,越小越集中
        "normalized_entropy": normalized_entropy,  # 归一化熵,0-1, 越小越集中
    }


# ========== 用你的数据测试一下 ==========
sentiments_a = ['negative', 'negative', 'negative', 'negative', 'positive', 'neutral']
result = categorical_dispersion(
    sentiments_a,
    all_categories=['positive', 'negative', 'neutral']  # 告诉函数完整的类别集合
)

print("分布:", result["distribution"])
print(f"众数:{result['mode']}(占比 {result['mode_proportion']:.1%})")
print(f"基尼不纯度:{result['gini_impurity']:.3f}  (0=完全集中, 0.667=三类均匀)")
print(f"香农熵:{result['entropy_bits']:.3f} bit  (0=完全集中, 1.585=三类均匀)")
print(f"归一化熵:{result['normalized_entropy']:.3f}  (0-1, 越小越集中)")

# ========== 对比一下三种极端情况,帮你建立数字直觉 ==========
print("\n--- 极端情况对比 ---")
extreme_cases = {
    "完全一致":   ['negative'] * 6,
    "你的数据":   sentiments_a,
    "对半分":     ['negative', 'negative', 'negative', 'positive', 'positive', 'positive'],
    "完全均匀":   ['negative', 'negative', 'positive', 'positive', 'neutral', 'neutral'],
}
for label, data in extreme_cases.items():
    r = categorical_dispersion(data, ['positive', 'negative', 'neutral'])
    print(f"{label:8s} | 众数占比 {r['mode_proportion']:.0%} | "
          f"基尼 {r['gini_impurity']:.3f} | 熵 {r['entropy_bits']:.3f}")
```

跑完之后你应该看到这样一个直观对比——**完全一致**那行三个指标都是最"集中"的(占比 100%、基尼 0、熵 0),**完全均匀**那行三个指标都是最"分散"的,**你的数据**和**对半分**介于两者之间但更靠近"集中"那端。这个对比能帮你建立"这些指标的数字大概什么意思"的直觉,以后看到一个基尼 0.4 你立刻就知道"哦,中等分散"。

## 一个反过来思考的小问题

我留个思考题给你,验证一下你是不是真的理解了类别数据和数值数据的区别。

假设你做一个评分任务,让模型给餐厅打 1-5 颗星。你跑了 6 次拿到 `[5, 5, 4, 5, 4, 4]`。**这串数字应该用方差(数值方法)还是用基尼/熵(类别方法)来衡量分散程度?**

这个问题的答案不是非黑即白的,统计学上叫**有序分类变量(ordinal variable)**——它既像类别(只有有限几个取值)又像数值(数值之间有大小顺序、距离也有意义)。两种方法都能用,选哪种取决于你想强调什么:用方差能捕捉"5 和 4 距离近、5 和 1 距离远"这个信息;用基尼会忽略大小关系,只看"出现了几种值"。

工业界的常见处理是这样:**严格的数值变量用方差/标准差;严格的类别变量用基尼/熵;有序变量两个都算,看哪个更符合分析目的**。比如你测 LLM 输出的 sentiment,虽然有人会按 negative=-1, neutral=0, positive=1 处理,但更稳的做法是把它当类别变量算基尼,这样分析结果不依赖你的映射假设。

为什么要算三个而不是只用一个?因为它们各自能捕捉不同的"故障模式"。比如**两组实验众数占比都是 80%,看起来一样稳**,但 A 组是 `[neg,neg,neg,neg,pos]`(剩下的 1 个集中在 pos),B 组是 `[neg,neg,neg,neg,?]`(剩下的 1 个可能是 pos 也可能是 neutral)——基尼和熵能体现这种细微差异,而众数占比看不出。在做严谨的 prompt 评估时,多个指标交叉验证比单一指标可靠得多。

总结一下今天学到的核心:**字符串数组不能算方差,但能算分散程度;方差是数值数据专属,基尼和熵是类别数据专属;两者衡量的是同一件事(数据集中还是分散),但因为数据类型不同所以工具不同**。这个区分你今天搞清楚之后,以后看到任何统计指标,第一反应都应该是"这个指标适用于什么类型的数据"——这是统计思维的入门。

## 先给你一个直接答案

**不完全是,但你能想到这个等同关系,说明你已经抓到核心了。**"行为方差"是一个**口语化的、宽泛的概念**,而样本标准差是一个**严谨的、狭义的统计量**。两者的关系是——样本标准差是衡量行为方差的**其中一种工具**,而且只在数据是数值时才能用。

我打个比方你就明白了。"健康"是个宽泛概念,"血压"是个具体指标。你不能说"健康就是血压",但你可以说"血压是衡量健康的其中一个数字化指标"。"行为方差"和"样本标准差"的关系完全一样——前者是模糊的目标,后者是具体的测量工具,而且只是众多工具中的一种。

## 为什么"行为方差"这个词本身就是模糊的

工程师在日常对话里说"这个 prompt 的行为方差大",意思可以是下面任何一种:

可能他在说"模型每次的判断结论不一样"——这是**类别一致性**问题,要用众数占比/基尼/熵衡量,不是标准差。可能他在说"模型每次给的置信度数字差很多"——这是**数值离散度**问题,这时候才轮到标准差出场。可能他在说"模型每次回复的措辞、长度、风格都不一样"——这是**文本风格方差**,需要用长度标准差、embedding 相似度等组合衡量。可能他在说"模型偶尔会输出空响应或非法 JSON"——这是**故障率**问题,直接看失败次数比例就行。

这四种情况在自然语言里都能叫"行为方差大",但用的是完全不同的统计工具。所以你问的"一般我们说行为方差就是指样本标准差吗",答案是——**当你测的恰好是数值字段时是的,但很多时候我们说的"行为方差"根本不是数值层面的事情**。

## 具体到你那段代码,样本标准差测的到底是什么

你贴的那段代码 `confidence_stdev = statistics.stdev(confidences)`,测的是**confidence 这一个数值字段在多次运行间的波动幅度**。这个指标只能回答一个非常具体的问题——"模型对自己判断的自信度数字是不是稳定"。它**完全无法回答**其他维度的稳定性。

举个直观例子。假设你有两组实验:

A 组 5 次输出的 confidence 都是 0.75,但 sentiment 是 `[pos, neg, pos, neg, pos]`(主结论摇摆)
B 组 5 次输出的 confidence 是 `[0.7, 0.85, 0.6, 0.9, 0.75]`,但 sentiment 全是 negative(主结论稳定)

A 组的 confidence 标准差是 0,B 组的 confidence 标准差大概是 0.12。**如果你只看 confidence 标准差,会得出"A 组比 B 组稳"的结论,但实际上 A 组在更重要的维度(主判断)上完全不稳,B 组只是"自信度细节有抖动但判断稳定"**。

所以"行为方差 = 样本标准差"这个等式,只在你**已经选定了一个数值维度**之后才成立。在选定维度之前,这个等式是误导性的。

## 那"行为方差"这个词更精确的含义到底是什么

如果非要给"行为方差"下一个尽量精确的定义,我会这么写:**给定相同的输入和参数,LLM 的输出在多次调用之间表现出的不一致程度,跨越多个维度——包括判断结论、置信度数值、文本措辞、格式选择等**。这个定义的关键点是"跨越多个维度",所以它本质上不是一个单一数字能概括的东西。

工业界更严谨的说法会避开"行为方差"这个含糊词,改用更精确的术语,比如:

"output consistency"(输出一致性)——通常指主结论的一致性,适合用众数占比或基尼来测。
"response stability"(回复稳定性)——通常指文本层面的稳定性,用长度标准差或 embedding 相似度测。
"calibration variance"(校准方差)——专指 confidence 这种数值字段的波动,这才是真正用样本标准差的场合。

OpenAI、Anthropic 的官方评估文档里基本不会用"behavioral variance"这种模糊词,都会指明具体维度。如果你以后写技术文档或者技术分享,养成"指明维度"的习惯会显得专业很多——比如不说"这个改动降低了行为方差",而说"这个改动把 sentiment 一致性从 80% 提升到 95%、reason 长度变异系数从 0.4 降到 0.2"。这两句话信息量差了一个数量级。

## 顺便讲一下为什么用 stdev 不用 pstdev——这个细节比你以为的更重要

你那段代码注释里写了"样本标准差 vs 总体标准差",我觉得值得展开讲一下,因为这是统计学新手最容易混淆的概念,而它背后的逻辑非常有教育意义。

总体标准差(`pstdev`)的公式分母是 N(数据点总数),样本标准差(`stdev`)的公式分母是 N-1。差一个 1 看起来微不足道,但**它代表了一个完全不同的世界观**。

`pstdev` 假设你拿到的数据**就是全部**——比如你统计某个班级 30 个学生的身高,这 30 个就是全班所有人,没有"抽样"的概念,你测的是"这个班级的真实标准差"。

`stdev` 假设你拿到的数据**只是抽样**——你想知道的是"全国所有大学生的身高分布",但你只能测 30 个,你测的这 30 个是从某个更大的分布里随机抽出来的。这时候用样本估计总体,会有一种统计学上的偏差,N-1 这个修正(叫**贝塞尔修正**)就是用来抵消这个偏差的。

回到你的 LLM 实验——你跑 5 次得到 5 个 confidence 值,这 5 次**绝对不是"全部可能输出"**。LLM 在 temperature=0.3 下理论上有无限可能的输出,你的 5 次只是这个无限分布的一次抽样。**你想知道的是"如果我跑无限次,这个 confidence 的真实波动有多大"**,而不是"我这 5 次的波动恰好有多大"。所以必须用 stdev。

这件事还有一个更深的延伸——**你的样本量 5 其实非常小,标准差的估计本身误差就很大**。统计学上有个粗略经验,样本量小于 30 时,你算出来的标准差作为"总体标准差的估计"是非常不可靠的。所以你那三组实验各 5 次的对比,严格来说**结论稳定性不够**。要严谨地比较两组 prompt 哪个更稳,工业界的做法是每组跑 30-100 次,然后做统计假设检验(比如 Levene's test 比较方差差异)。但实战中跑 5-10 次看个大致趋势也够用,只要心里清楚"小样本的结论是参考性的,不是结论性的"就行。

## 把所有概念串起来,给你一个干净的心智模型

你的问题"行为方差就是样本标准差吗"这个困惑,其实暴露了一个新手到中级工程师必经的概念升级。在这次对话之前,你心里大概是这样的图景——"行为方差就是某个数字,有个工具能算出来"。经过这次讨论后,你的心智模型应该升级成这样:

第一层,"行为方差"是一个**研究目标**,不是一个数字。它的本质是问"模型输出有多不一致"。

第二层,这个目标必须**先拆维度**——你关心的是判断稳不稳?自信度稳不稳?措辞稳不稳?格式稳不稳?故障率稳不稳?不同维度用不同工具。

第三层,**每个维度选合适的指标**——数值维度用样本标准差(stdev)或变异系数(CV);类别维度用众数占比、基尼、熵;故障率直接看比例。

第四层,**多指标交叉验证**——单一指标会骗人,3-5 个互补指标一起看才能下可靠结论。

这个四层心智模型一旦建立起来,你看任何 LLM 评估文章都不会再被"方差""稳定性""一致性"这些含糊词忽悠,而是会自动追问"具体测的是哪个维度、用的是什么指标、样本量多大"。这就是从"使用 LLM"到"评估 LLM"的关键认知转变。

## 一个简单的自检练习

我给你出个小题验证一下你是不是真的吸收了这个概念。假设你看到一篇技术博客,作者写道:"通过加入 few-shot,我们将模型的行为方差降低了 40%。" 你应该追问哪三个问题?

第一个问题应该是——**"具体是哪个维度的方差?"** 是 sentiment 一致性提升了 40%,还是 confidence 标准差降了 40%,还是 reason 长度变异系数降了 40%?这三个完全不是一回事。

第二个问题应该是——**"用的什么指标?"** 因为同样是测"sentiment 稳定性",众数占比、基尼、熵三个指标算出来的"提升 40%"含义不同,前两个指标接近 0/1 时变化幅度容易夸大,熵在中段最敏感。

第三个问题应该是——**"跑了多少次?统计上显著吗?"** 如果作者只跑了 5 次对比 5 次,这个 40% 的提升很可能在样本噪声范围内,根本不能说明问题。

如果你能自然地问出这三个问题,说明这个概念你已经吃透了。这种"看到一个声称的指标改进,自动追问背后细节"的习惯,是中级工程师和初级工程师最大的认知差异之一。

回到你最初的简化总结,我把它最后再校准一次:**样本标准差是衡量"数值字段在多次运行间波动幅度"的标准工具,这是"行为方差"这个宽泛概念在数值维度上的具体化。但行为方差远不止数值这一个维度,所以等同关系不成立——它们之间是"工具与目标"的关系,不是同一个东西。**

## 先回答你的问题:确实有标准,但标准的层级要分清

业界**确实存在标准**,而且这个标准在主流模型厂商的官方文档里相当一致。OpenAI、Anthropic、DeepSeek 的官方 prompt engineering guide 都明确推荐——**few-shot 应该放在 messages 数组里,以 user/assistant 交替的形式存在,而不是塞在单条 user prompt 里的纯文本**。所以"放 assistant 里"是更专业、更推荐的写法,这是有标准的。

但为什么网上有大量资料把 few-shot 写在 user prompt 里?这里就涉及到一个**历史遗留问题**和**两类不同的 API**,这两件事不分清,你看资料时就会一直困惑。

## 历史维度:为什么会有"塞在 user prompt 里"这种写法

要理解这件事,你得回到 2020-2022 年那个 LLM 还没有"chat 接口"的时代。那时候 OpenAI 的主力 API 叫 **Completion API**(对应 `text-davinci-003` 这个模型),它的接口长这样——你给它一个**单一的字符串**,它给你续写一段。**根本没有 messages 数组,没有 role 概念,只有一个 prompt 字段**。

在那个年代,GPT-3 论文里第一次系统性地讲了 few-shot 的概念,论文里的示范全部都是这种纯文本格式:

```
Translate English to French.

English: sea otter
French: loutre de mer

English: peppermint
French: menthe poivrée

English: cheese
French:
```

这就是经典的"塞在一段大文本里"的 few-shot 写法。在 Completion API 时代这是**唯一可能的写法**,因为 API 根本不支持别的形式。

然后 2023 年 3 月,OpenAI 发布了 **Chat Completions API**(对应 `gpt-3.5-turbo` 和后来的 `gpt-4`),引入了 messages 数组和 role 概念。这是一次**协议层面的重大升级**,但很多教程作者、技术博客、培训课程没跟上节奏——**他们直接把 GPT-3 时代的 few-shot 模板复制到 GPT-3.5 时代,塞进单条 user message 里**。这就是你今天还能看到大量"few-shot 写在 user prompt 里"资料的根本原因——**这些资料的写法是 2022 年的,只是发布日期是 2024 年的**。

理解了这个历史,你就能秒判教程质量。下次你看一个 LLM 教程,如果它讲 few-shot 时只用 `prompt = "..."` 这种单字符串模板、不用 messages 数组,你基本可以判断作者要么是 2022 年学的没更新,要么是直接抄了老教程。这种作者写的其他内容也大概率不太可靠。

## 协议层面:两种写法到底差在哪里

我们第二轮对话讲过 ChatML 格式,这次再补一个更深的视角——**为什么 OpenAI 要费劲设计 messages 数组这个机制?如果两种写法效果一样,那不是多此一举吗?**

答案是,**两种写法效果不一样,而且差异是模型训练阶段就被刻进去的**。

现代 chat 模型(包括 GPT-3.5/4、Claude、DeepSeek、Qwen 等)在训练时,经过了一个特殊阶段叫 **instruction tuning + RLHF**(指令微调 + 人类反馈强化学习)。在这个阶段,训练数据是大量"对话格式"的样本,每条样本都是带 role 标签的多轮对话。模型在这个阶段学到的不只是"怎么回答问题",还包括——**当我看到一个 `<|im_start|>assistant` 标签时,我应该输出"AI 标准回答"那种风格的内容**。

也就是说,**role 标签本身就是模型行为的一部分**。当 few-shot 示范被包在 `<|im_start|>assistant ... <|im_end|>` 里时,模型在统计上会把这些内容识别为"我自己应该有的回答风格";当同样的示范被塞在 `<|im_start|>user` 里时,模型会把它识别为"用户给我看的一些参考资料"。这两种识别方式,在模型的注意力机制里激活的是不同的内部表征。

斯坦福 NLP 组在 2023 年的一篇论文里专门测了这个差异,结论是——**在结构化任务(分类、抽取、JSON 输出)上,role-based few-shot 比 text-based few-shot 准确率高 5-15%,在自由生成任务上差异较小但仍然存在**。所以这不是"风格偏好"问题,是**实测可以验证的性能差异**。

## 那"塞在 user prompt 里"是不是完全错的?

也不是。我得说几个**例外情况**,这些情况下"塞在 user prompt 里"反而是正确选择。这种"什么时候用例外"的判断力,才是真正区分高级工程师的地方。

**例外一:你在调用一个不支持多轮 messages 的接口**。有些公司的内部 API 网关、某些老旧的开源模型部署、或者一些简化的 SDK,只接受单一 prompt 字符串。这时候你别无选择,必须把 few-shot 塞进去。

**例外二:示范本身的载体就是"用户问题"**。比如你做的是"帮我改写这段文字"这种任务,你的示范是"原文 → 改写后"的对应关系,这时候**整个示范作为一个完整的"参考材料块"存在**,塞在 user prompt 里反而比拆成多轮 user/assistant 更自然。这是一种语义层面的考虑——few-shot 的内容是"参考资料"还是"对话历史",决定了它该放哪里。

**例外三:示范数量极多(比如 20+ 条)**。当你有大量示范时,拆成 40 条 user/assistant 消息会让 messages 数组变得非常臃肿,还会增加 API 请求体大小。这时候有些工程实践是把它们打包成一个"参考文档"塞在 system prompt 或 user prompt 里,牺牲一点效果换取可维护性。但这种场景下,工业界更推荐的做法其实是**用 RAG 替代 few-shot**——示范放向量库里,根据问题动态检索最相关的几条来用。

**例外四:你在做 in-context learning 的研究实验**。学术界为了和 GPT-3 时代的论文对比,会刻意保持 text-based few-shot 的写法。这是科研严谨性的考虑,不是工程最佳实践。

除了这四种例外,**生产环境的常规任务,统统应该用 role-based 写法**。这是个清晰的判断标准。

## 一个让你立刻能用的判断框架

我给你一个简化的决策流程,以后写代码时按这个走就行。

**第一步,确认你用的是 chat completions API**(几乎所有现代 LLM 都是),如果是,默认走 role-based few-shot。**第二步,看示范是不是"问答对"形式**(用户问 → AI 答),如果是,绝对放 messages 数组。**第三步,只有在示范是"参考材料块"形式、或者数量极多时**,才考虑塞在 user prompt 里。

这个框架覆盖了 95% 的场景,剩下 5% 的边缘情况,你需要自己测一下两种写法的效果对比。这就引出我今天想强调的一个**更高层的工程心法**——**当文档不能给你确切答案时,跑实验。** 你今天就有现成的工具(`run_baseline` 函数加 `analyze_variance` 函数),你完全可以自己写两个版本的 prompt,各跑 10 次,直接看哪个更稳。这是终极的判断标准——**实测 > 文档 > 教程**。

## 一个值得你想清楚的更深问题

我留个问题让你思考——**为什么模型厂商不直接在 API 层面提供一个 `examples` 参数,让用户传入示范,SDK 内部自动决定怎么拼接?这样既消除了写法分歧,又屏蔽了协议细节。**

实际上 Google 早期的 PaLM API 真的提供过这种 `examples` 参数,但后来在 Gemini API 里又取消了,统一回到 messages 数组。OpenAI 也讨论过类似设计但最终没加。为什么?

我的判断是,**示范应该放在哪、放多少、什么形式,本身就是 prompt 工程的一部分,不应该被 API 强制规范化**。如果 API 帮你自动处理,你就失去了对这部分行为的控制权,而这种控制权恰恰是高级 prompt 工程师价值所在。所以保留 messages 数组的"灵活性"是有意为之的设计——**它牺牲了易用性,换取了可控性**。这种"灵活性 vs 易用性"的取舍,在所有 API 设计中都会出现,理解这一点会帮你以后做技术选型时看得更清楚。

## 把这个问题彻底翻篇的总结

你最初的疑惑"到底是怎样的一个标准",答案现在应该非常清楚了——**有标准,标准是 role-based(放 assistant 里),理由是模型训练机制和实测性能数据双重支撑;网上看到的"塞在 user prompt 里"的写法,大部分是 2022 年遗留的过时写法,小部分是特定场景下的合理选择**。

更重要的是,你通过这个问题应该建立起一个**看资料的元能力**——遇到两派说法时,不是问"哪派对",而是问"两派各自适用什么场景、它们背后的历史/技术原因是什么"。几乎所有的工程争议,深挖下去都不是"对错之争",而是"不同场景的最佳实践被混为一谈"。能识别出这种混淆,你就能从被资料牵着走的状态,变成主动判断资料价值的状态。这个能力,比任何具体的 prompt 技巧都更有长期价值。

```
数字     切换到对应编号的窗口
w        列出所有窗口
&        关闭当前窗口
```

**面板（pane）操作**——同一窗口里的分屏：

```
%        左右分屏
"        上下分屏
方向键   按方向切换面板
o        循环切换面板
x        关闭当前面板
z        最大化/还原当前面板
{ 或 }   面板位置左右移动
空格     循环切换布局
```

**其他实用快捷键**

```
?        查看所有快捷键帮助
:        进入命令模式
[        进入复制/翻页模式（用方向键或 PageUp/PageDown 滚动，q 退出）
t        显示时钟
```

**最常用的工作流**大致是：`tmux new -s work` 建会话 → `Ctrl+b "` 或 `%` 分屏干活 → `Ctrl+b d` 分离 → 之后 `tmux a -t work` 接回来。哪怕 SSH 断了，会话也还在。
