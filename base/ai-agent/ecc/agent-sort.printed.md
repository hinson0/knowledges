# Agent Sort 技能

## 定位

为特定仓库构建**证据驱动**的 ECC 安装方案。核心问题：全量安装 ECC 太嘈杂，需要裁剪到项目实际需要的子集。

## 核心概念：两桶分类

| 桶 | 含义 | 加载方式 |
|---|---|---|
| **DAILY** | 每次会话自动加载 | 安装到 `.claude/skills/` 等活跃路径 |
| **LIBRARY** | 保留但不默认加载 | 通过搜索、路由技能或手动使用访问 |

关键区分：LIBRARY ≠ 删除，而是"不占上下文窗口但随时可查"。

## 分类决策逻辑

**提升为 DAILY：**
- 仓库明确使用匹配技术栈（有对应文件扩展名、配置、依赖）
- 组件足够通用，每次会话都有帮助
- 仓库已依赖相应运行时或工作流

**降级为 LIBRARY：**
- 组件不在当前技术栈上（如 Python 仓库里的 Rust 规则）
- 将来可能需要但不是每天需要
- 增加上下文开销却没有即时价值

## 证据来源（必须基于仓库本地数据）

- 文件扩展名分布（`rg --files`）
- 包管理器和锁文件（package.json / pyproject.toml / Cargo.toml / go.mod）
- 框架配置文件（next.config.ts / django settings 等）
- CI 和钩子配置
- 构建/测试脚本
- import 语句和依赖清单

## 6 个并行审查轮次

可用子代理时并行，否则串行：

1. **代理** — `agents/*`
2. **技能** — `skills/*`
3. **命令** — `commands/*`
4. **规则** — `rules/*`
5. **钩子和脚本** — 钩子表面、MCP 健康检查、辅助脚本、OS 兼容性
6. **扩展** — 上下文、示例、MCP 配置、模板、指导文档

## 工作流（5 步）

1. **读取仓库** — 确认真实技术栈（语言、框架、包管理器、测试栈、部署表面）
2. **构建证据表** — 每个组件记录：路径、类型、建议桶、证据、理由
3. **DAILY/LIBRARY 决策** — 基于证据表做分类
4. **构建安装方案** — 将分类转化为具体安装/移除/路由操作
5. **验证** — 确认 DAILY 文件存在、过时规则已清理、不兼容钩子未安装

## 可选：库路由器

创建 `.claude/skills/skill-library/SKILL.md` 作为搜索入口，包含：
- DAILY vs LIBRARY 的说明
- 分组触发关键词
- 库引用位置（不复制完整技能内容）

## 后续交接

| 场景 | 交接目标 |
|---|---|
| 交互式安装或修复 | `configure-ecc` |
| 重叠清理或目录审查 | `skill-stocktake` |
| 更广泛的上下文精简 | `strategic-compact` |

## 输出格式

按顺序返回：技术栈摘要 → DAILY 清单（带证据）→ LIBRARY 清单（带证据）→ 安装方案 → 验证报告

## 不可违反的规则

- 以当前仓库为唯一事实来源
- 每个 DAILY 决策必须引用具体仓库证据
- 不安装仓库无法使用的钩子/规则/脚本
- 优先 ECC 原生机制，不引入第二套安装系统
