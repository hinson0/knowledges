## Overhead（额外开销）

**Overhead** 在编程和系统里指的是 **"额外开销"**——不是干正事本身，而是为了支撑系统运作而额外消耗的资源。比如时间、内存、网络请求、金钱成本等。

### 在 AI Agent 场景下，overhead 通常指

- **Token 开销**：Prompt 里塞了太多背景信息，或多次重复调用模型，导致不必要的 Token 消耗（直接等于成本）。
- **延迟开销**：多一次工具调用、多一轮推理循环，响应时间就多一截。
- **工程开销**：引入复杂框架（如一些过度封装的库）带来的维护、调试、学习成本。
- **存储/检索开销**：向量数据库的建库、查询耗时，以及为了保持上下文而做的 Checkpoint 读写。

### 举个例子

你的 Agent 每次都要先检索知识库，再生成回答。其中"检索"这一步并不直接产生答案，它就是 **overhead**（为了答得更准而付出的额外代价）。

优化时，我们会用缓存（如语义缓存）来减少这种 overhead，让下次类似问题直接返回答案，省掉重复的检索和推理成本。

> **简单记：干正事外的所有成本，都是 overhead。**

## Eager Mode（早停 / 即时执行）

"Eager mode"通常被称为"急切模式"或"即时执行模式"，指的是一种"所见即所得"的任务处理方式。当程序执行到一行操作代码时，系统会立刻计算并返回结果，而不是先积压起来等待后续统一处理。

### 核心思想

它的核心思想是 **立即执行**（Immediate execution）。Eager Mode 遵循我们日常写 Python 脚本的习惯：代码写到哪里，程序就执行到哪里，并立刻返回结果。这与另一种"延迟执行"（Lazy mode）模式完全不同，后者会先构建一个完整的计算蓝图，再寻找最佳时机统一执行。

### 典型应用场景

为了更立体地理解，我们可以看几个典型的应用场景，它们都用 Eager 实现了"即时响应"：

- **💻 深度学习框架**
  - **核心机制**：`操作 = 立即在 GPU 上执行`，支持使用 Python 原生控制流（如 `if/else`）。
  - **解决问题**：牺牲一点性能上限，换来 **极致的开发和调试灵活性**，深受研究人员喜爱。
- **🧠 大模型推理 (vLLM)**
  - **核心机制**：每次都完整调用模型，根据输入 **动态计算** 参数和内存，不固化。
  - **解决问题**：应对模型结构的复杂性，是 **兼容性最强、最不容易出错** 的后备方案。
- **🗣️ 语音/对话智能体**
  - **核心机制**：不等结束就根据中等置信度的"疑似结束"信号，**提前** 准备响应。
  - **解决问题**：利用生成响应的计算时间与用户停顿时长 **重叠**，大幅降低端到端延迟。
- **⚙️ AI 代码生成**
  - **核心机制**：模型边写出代码片段，边 **拿去做并行验证/执行**，提前暴露错误。
  - **解决问题**：打破生成-执行的串行瓶颈，抢占宝贵的计算资源来 **隐藏执行延迟**，显著提升效率。

### Eager vs Graph(Lazy)：灵活 vs 性能

在深度学习和推理部署中，常把 Eager 和 Graph(Lazy) 相比较：

| 维度         | Eager Mode（即时执行）                              | Graph Mode（Lazy / 静态图）                                 |
| ------------ | --------------------------------------------------- | ----------------------------------------------------------- |
| **处理方式** | 边定义，边执行，代码逐行运行并立即返回具体值        | 先整体定义计算蓝图，再统一编译执行                          |
| **调试能力** | 简单直观，完美支持 `pdb`，可随时检查中间变量        | 困难，流程封闭不透明，难以追踪内部细节                      |
| **灵活性**   | 极高，支持动态控制流，便于模型创新                  | 低，蓝图一旦定义就难以改动，对复杂控制流不友好              |
| **运行效率** | 较低，每次操作都要调用一次 GPU，CPU-GPU 通信开销大  | 极高，可对蓝图全局优化（算子融合、内存复用），CPU 开销极小  |

### 大模型部署与推理：`enforce_eager` 参数

在大模型部署框架（如 `vLLM`、`SGLang`）中，**`enforce_eager`** 是关键参数，通常默认为 `False`：

- **`enforce_eager=False`（推荐生产）**
  - 启用 **CUDA Graph** 进行极致加速。
  - **原理**：捕获一次模型执行序列并固化成图，后续请求只需"重放"此图来跳过重复的 CPU 指令开销。
  - **优缺点**：吞吐量极大，但需额外显存，且要求模型结构必须是"静止"的；如果模型有动态控制流可能会报错。

- **`enforce_eager=True`（调试/兼容）**
  - 走 **纯 Eager 模式**，彻底关掉 CUDA Graph。
  - **原理**：每次请求都老老实实跑一遍完整模型。
  - **优缺点**：兼容性最好，应对复杂动态模型或调试的唯一选择；CPU 开销大，性能最低。

### AI Agent 的"早停"机制（Speculative Early Exit）

在 AI Agent 的推理场景下，"早停"（Early Stopping/Early Exit）是一种动态优化策略，与思考效率直接相关。

- **核心问题**：大模型推理时的冗长或无效思考（如思维链、Dialogue Turns）会带来巨大计算浪费和响应延迟。
- **早停机制**：在推理过程中，动态判断"思考价值"来决定是否提前跳出，从而节约计算。
- **实践突破（如 SPECEXIT）**：结合投机采样（Speculative Decoding），在保持任务效果无损的前提下，将模型思维链长度缩短 66%，可实现端到端 2.5 倍加速。

> 你目前大模型推理中用得多的是哪类？如果主要是为了快速验证想法和 debug，可以试试在推理参数里把 `enforce_eager` 设成 `True` 来跑。要是你有一些更具体的场景（比如语音助手或代码生成），也可以告诉我，我再结合你的需求给你详细建议。

## ReAct 模式

ReAct 是构建 AI Agent 的核心思维范式，全称 **Reasoning + Acting**（推理与行动协同）。它让 Agent 不再是"一次调用模型直接出答案"，而是 **在推理和行动之间循环迭代，直到解决问题**。

### 一句话理解 ReAct

> **一边想，一边做，做错了就往回修正，直到做完。**

- **传统 LLM 调用**：你给我问题，我给你最终答案。
- **ReAct 模式**：你给我问题，我 **先思考下一步该干嘛**，然后 **立刻执行那个动作**，得到反馈后 **再思考下一步**，如此循环。

### 核心循环

```
用户提问 → 推理（Thought） → 行动（Action） → 观察（Observation） → 推理 → 行动 → …… → 最终答案
```

一个完整的 ReAct 循环包含三个角色：

| 组件                    | 作用                                                                                               |
| ----------------------- | -------------------------------------------------------------------------------------------------- |
| **Thought（思考）**     | 对当前状态的分析和下一步决策。Agent 会自言自语："我需要先查天气"、"这个 SQL 报错了，我改一下条件试试" |
| **Action（行动）**      | 实际调用工具、执行代码、查询数据库、搜索网页等具体操作                                             |
| **Observation（观察）** | 行动执行后返回的结果（成功/失败/数据），作为下一轮思考的输入                                       |

**终止条件**：当 Agent 的思考判断"我已经有足够信息回答用户了"，就停止循环，输出最终答案。

### 为何需要 ReAct

纯 LLM 有两个致命缺陷，ReAct 专门解决它们：

1. **知识截止与幻觉**
   模型训练数据是过去的，面对实时信息（如今天天气）就容易编造。ReAct 允许它 **外挂行动**：先调用天气 API 获取真实数据，再基于事实回答。

2. **复杂多步推理能力弱**
   对于"先查 A、再根据 A 的结果查 B，最后汇总"这类逻辑链，单次模型调用容易出错。ReAct 让模型 **显式地分解任务**，每一步都看到上一轮的结果，修正计划，步进完成。

### 在 LangGraph 中如何落地

LangGraph 天然是为 ReAct 而生的框架。你之前的 SQL 助手项目就是典型的 ReAct 实现：

```
       ┌──── 用户问题 ────┐
       │                  ▼
       │          ┌───────────┐
       │          │  agent    │ ← 思考：该查什么？怎么查？
       │          └─────┬─────┘
       │                │
       │     ┌──────────┴──────────┐
       │     │ tools_condition     │  ← 需不需要行动？
       │     └────┬───────┬────────┘
       │          │       │
       │   有工具调用    无工具调用
       │          │       │
       │          ▼       ▼
       │    ┌──────────┐  END（输出答案）
       │    │  tools   │  ← 行动：执行 SQL
       │    └────┬─────┘
       │         │  观察：返回查询结果
       └─────────┘
           循环回到 agent，根据结果继续推理
```

图中关键点：

- **`agent` 节点** → 负责"Thought"，由 LLM 生成（可能包含一个工具调用请求）。
- **`tools_condition`** → LangGraph 内置函数，检查最后一条消息是否包含 `tool_calls`。有则路由到 `tools` 节点，没有则结束。
- **`tools` 节点** → 负责"Action"，真正执行工具函数，并将返回结果作为消息写入状态。
- **自动循环** → `tools` 节点执行完后，边（edge）回到 `agent`，让 LLM 看到 Observation 后继续推理。

### 代码示例（LangGraph 0.4+）

以下是最精简的 ReAct Agent 实现框架，对应上面那个图：

```python
from typing import Annotated, TypedDict
from langgraph.graph import StateGraph, START, END
from langgraph.graph.message import add_messages
from langgraph.prebuilt import ToolNode, tools_condition
from langchain_openai import ChatOpenAI
from langchain_core.tools import tool
from langchain_core.messages import BaseMessage

# 1. 定义工具
@tool
def get_weather(city: str) -> str:
    """查询城市天气"""
    return f"{city} 当前 25°C，晴天"

# 2. 定义状态（共享内存）
class State(TypedDict):
    messages: Annotated[list[BaseMessage], add_messages]

# 3. 绑定工具到 LLM
tools = [get_weather]
llm = ChatOpenAI(model="gpt-4o-mini", temperature=0).bind_tools(tools)

# 4. 定义 agent 节点（推理步骤）
def agent_node(state: State):
    return {"messages": [llm.invoke(state["messages"])]}

# 5. 构建图
builder = StateGraph(State)

builder.add_node("agent", agent_node)
builder.add_node("tools", ToolNode(tools))

builder.add_edge(START, "agent")
builder.add_conditional_edges("agent", tools_condition, {
    "tools": "tools",
    END: END
})
builder.add_edge("tools", "agent")  # 行动后回到推理

graph = builder.compile(checkpointer=memory)
```

### 一次完整的 ReAct 对话过程

假设用户问："上海现在热吗？"

1. **第一次推理**
   `agent` 节点收到 `[HumanMessage("上海现在热吗？")]`，LLM 返回一条带有工具调用的 `AIMessage`：
   - `Thought: 我需要知道上海现在气温`
   - `ToolCall: get_weather("上海")`

2. **条件路由**
   `tools_condition` 检测到有工具调用，路由到 `tools` 节点。

3. **行动 & 观察**
   `tools` 节点执行 `get_weather("上海")`，返回一个 `ToolMessage`，内容是 `"上海 当前 25°C，晴天"`。

4. **第二次推理**
   回到 `agent` 节点，LLM 看到所有历史消息（包含工具返回的温度），推理：
   - `Thought: 25°C 属于舒适，不热`
   - 然后生成最终回答给用户，此时不再包含工具调用。

5. **再次条件路由**
   `tools_condition` 发现没有工具调用，进入 `END`，把答案返回给用户。

整个过程自动循环，直到 LLM 觉得"我不需要再做事了"为止。

### ReAct vs 纯 Chain

| 维度     | 纯 Chain              | ReAct Agent                          |
| -------- | --------------------- | ------------------------------------ |
| 流程     | 线性，A→B→C，固定路径 | 动态，有分支和循环                   |
| 决策     | 预定义，无自主权      | LLM 根据上下文实时决定下一步         |
| 错误恢复 | 困难，需人工写分支    | 天然可自我纠错（看到错误后重新推理） |
| 复杂度   | 简单，适合明确任务    | 适合开放式、多步骤、需工具的场景     |

### 实战中的 ReAct 调优

在你后续的实战中，有几个关键的优化点：

- **Prompt 设计**：在系统提示里明确告诉模型"只有当确实需要时才调用工具，不要胡乱猜测"，可以减少不必要的工具调用（降低 overhead）。
- **Observation 格式化**：工具返回的结果要简洁、结构化（如 JSON），让 LLM 容易消化。如果返回一个 5000 行的网页 HTML，Agent 就推理不动了。
- **最大循环次数控制**：在图编译时设置 `recursion_limit` 或实现一个计数器节点，防止 Agent 进入死循环（不断调同一个工具但得不到新信息）。
- **LangSmith / Langfuse 追踪**：把每一次 Thought → Action → Observation 链记录下来，你就能直观看到 Agent 哪里"犹豫"了、哪里"犯傻"了，进而优化 Prompt 和工具。

## Few-shot

**Few-shot** 是在提示词里给模型 **提供几个范例**，让它照着格式和逻辑去模仿，从而稳定输出。

### 简单类比

你第一次做 Excel 透视图，看了三个操作示例后，照着做就能做对——**那几个示例就是 Few-shot**。

### 在优化场景下的用法

表格里提到"Few-shot examples"，意思是：在 system prompt 中直接给一个具体例子：

```
示例：
用户：在 /tmp/test 目录下创建一个文件 hello.txt
助手：直接调用 write_file("/tmp/test/hello.txt", "content")，不先调用 run_shell("mkdir /tmp/test")
```

模型看到这个范例后，就会 **稳定模仿**，不再每次都试探性先建目录。

### Few-shot vs 纯描述

- **纯描述（低效）**："不要先用 mkdir，直接用 write_file"
- **Few-shot（高效）**："你看，遇到这种情况，就该这么做：<示例>"

这和你在 Day 2 写的"客服意图分类器"里的 Few-shot 完全一样——给几个 `用户: "..." → 结果: "..."` 的例子，模型就懂了。

### 词源

- **few**：少量的、几个的
- **shot**：尝试、一次机会（此处可理解为"示例"或"样本"）

组合起来就是 **"少量示例"** 或 **"只给几个例子"**。

在机器学习里，**"shot"** 是早期从 "one-shot learning"（一次学习）这类术语延续下来的说法，表示模型从几个标注样本中学习完成任务。日常可简单记成：**"给你几个例子学一下，马上照做"**。

> **简单记：描述一千遍，不如演示一遍。**
