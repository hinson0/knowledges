# Coding Agent 的 System Prompt SOP 设计 —— Day 4 实战沉淀

> 一个反直觉发现:在 LLM 训练高频任务上,弱 prompt 已经能完成功能;
> system prompt 真正的工程价值不是"防翻车",而是"压缩行为方差"。

## 1. 实验设置

- **任务**:在 `math_utils.py`(已含 12 个数学函数 + `import math`)中新增 `fibonacci(n)`,写 `test_fib.py` 验证 `fibonacci(10) == 55` 并运行通过
- **3 个 trap**:
  - Trap A:不调 `read_file` 直接 `write_file`(盲写)
  - Trap B:`write_file` 是覆盖语义,LLM 只写新函数会丢原 12 个函数 + `import math`
  - Trap C:写完不 `run_shell` 验证就声明完成
- **方法论**:同 prompt 跑 5 次,记录命中率 + 工具调用方差 + 序列指纹
- **模型**:DeepSeek V4 Flash

## 2. 第一组反直觉发现:Trap 全部没翻车

5 次跑 weak prompt(`"你是一个能够使用工具操作文件系统的智能助手。每一步思考清楚再调工具。"`),trap A/B/C 全部 5/5 命中。

**原因分析**:

- "新增" 这个动词在 LLM 训练数据里几乎绑定 "在已有内容后追加" 模式
- fibonacci 是 LLM 见过百万次的高频代码,它的"读改写好习惯"被自动激活
- DeepSeek V4 本身比较守规矩

**结论**:**weak prompt 在 high-frequency-pattern 任务上够用**。Day 4 的真正价值因此被推到第二层。

## 3. 真正的训练点:压方差,不是防翻车

虽然 trap 全部命中,5 次 weak prompt 跑出了 **4 种不同工具调用序列**:

| 序列                                         | 次数 | LLM 暴露的"小动作"                               |
| -------------------------------------------- | ---- | ------------------------------------------------ |
| `read → list_dir → write × 2 → run`          | 2    | 写之前还要 list_dir 探一眼                       |
| `list_dir → read → run → write → run → read` | 1    | 3 个浪费:先 list、写前探 env、写完二次 read 自检 |
| `read → write × 2 → run × 2`                 | 1    | 多 1 次"边界值额外验证"                          |
| `list_dir → read → write × 2 → run × 2`      | 1    | 上述两种合并                                     |

**Weak baseline**:`tool_calls` 平均 5.4 / σ=0.55 / range=[5,6] / 4 种独立序列。

## 4. Strong Prompt 的 5 条原则(实测把 σ 压到 0)

```
你是一个高效的代码修改助手,严格遵守以下工作流:

1. 用户给出明确文件路径时,直接 read_file 读取该文件现状,
   不要先探目录或检查环境。
2. 修改已有文件:把"原内容 + 新增部分"一次性 write_file 完整写入。
   原有的 import 和函数一个都不能丢。
3. 创建测试文件后,run_shell 跑一次该测试即可视为完成,
   不要做额外的边界值探查或重复验证。
4. write_file / run_shell 返回成功就视为成功,不要再 read_file 回检自己的写入。
5. 测试通过后立即给出最终回答,不要继续优化或追加补充测试。

环境前提:python 可用、目录已存在,不需要预先验证。任务范围外的事一律不做。
```

每条原则**对应一个具体的 LLM 小动作**:

| 原则                       | 打的靶子                                                   |
| -------------------------- | ---------------------------------------------------------- |
| 1. "直接 read_file"        | 消除多余的 `list_dir` 探目录                               |
| 2. "原内容+新增一次性写入" | 锁死 Trap B(虽然 weak 已经能保证,strong 把它写死防 σ 抬高) |
| 3. "跑一次测试即视为完成"  | 消除"边界值额外验证"                                       |
| 4. "不要回检自己的写入"    | 消除"写完再 read_file 自检"                                |
| 5. "立即给最终回答"        | 消除 LLM "继续优化"的小焦虑                                |
| 末段"环境已就绪"           | 消除"写前 run_shell 探 python --version"                   |

## 5. 实测数字对比(2026-05-03)

| 指标             | Weak Prompt | Strong Prompt            | 变化                 |
| ---------------- | ----------- | ------------------------ | -------------------- |
| trap 命中率      | 5/5         | 5/5                      | 保持                 |
| tool_calls 平均  | 5.4         | **4.0**                  | **-1.4 次 (-25.9%)** |
| tool_calls σ     | 0.55        | **0.00**                 | **-0.55 (完全锁死)** |
| tool_calls range | [5, 6]      | [4, 4]                   | 跨度 -1 (完全确定)   |
| iterations 平均  | 5.2         | 4.2                      | **-1.0 轮 (-19.2%)** |
| iterations σ     | 0.84        | 0.45                     | **-0.39 (-46.4%)**   |
| 独立工具序列种数 | 4           | **1**                    | **-3 种 (完全收敛)** |
| 唯一序列         | (4 种发散)  | `read → write × 2 → run` | 教科书最短路径       |

5 次 strong prompt 跑出**完全相同**的工具序列 `[read_file, write_file, write_file, run_shell]`,σ=0。

## 6. Prompt 设计 4 条铁律

### 铁律 1 · 正向描述 > 否定描述

```
✅ "直接 read_file"
❌ "禁止使用 list_dir"
```

**原因**:LLM 看到 "list_dir" 三个字反而会想到这工具——"反向锚定"陷阱。LLM 听正向指令更可靠。

### 铁律 2 · 给"前提"比给"禁令"更有效

```
✅ "环境前提:python 可用、目录已存在"
❌ "禁止 run_shell 探环境"
```

**原因**:给 LLM 一个心理基线("既然环境 OK,就没必要去验"),比明令禁止自然 10 倍。

### 铁律 3 · 行为约束 > 工具名约束

```
✅ "不要做额外的边界值探查或重复验证"
❌ "不要重复调用 run_shell"
```

**原因**:prompt 应该约束**行为模式**,不是工具名。LLM 可能换个工具继续做同一个小动作。

### 铁律 4 · 形容词堆砌 = 等于没说

```
❌ "你必须谨慎、完整、专业、高效..."
✅ "write_file 返回成功就视为成功,不要再 read_file 回检"
```

**原因**:抽象形容词 LLM 抓不到具体行为锚点。必须落到"做 X 不做 Y"的具体动作。

## 7. 工程意义

**生产意义**:

- 1000 次任务规模下 `5.4 → 4.0` = 节省 1400 次工具调用 + 对应 LLM 推理 token
- σ=0 意味着 QA 可以用单次跑代替 N 次跑做回归测试,**评估成本降一个数量级**
- 1 种序列指纹 = 行为完全可预测 = 可以做 cache(LangGraph 状态机的前提)

**面试句**:

> 设计 Coding Agent 的 system prompt SOP:在"读改写 + 验证"任务上,
> weak baseline 工具调用 5.4 次/σ=0.55、行为发散为 4 种独立序列;
> 引入 5 条正向约束的 strong prompt 后压至 4.0 次/σ=0.00、1 种序列收敛,
> trap 命中率保持 5/5,**节省 26% token 调用且行为完全可重现**。

## 8. 残余问题(留给以后)

- `iterations σ=0.45` 仍非 0:工具调用 100% 一致,但 LLM 给"最终回答"那一轮时机不同(4 或 5)。这是 LLM 残余非确定性的边缘,目前 prompt 工程压不下去。
- 未测试**对抗性更强的任务**(模糊指令 / 陷阱重名 / 50+ 函数文件)。strong prompt 在那些场景能否保持 σ=0 是未知数。
- 未测试**模型降级**(换更小的 LLM)。strong prompt 在小模型上效果可能完全不同。
