# 向量夹角与余弦相似度的几何与高维数学

> 来源：week2/day5_workspace/0510_2/1815.md。线性代数标准概念。从二维直观出发，扩展到 1024 维 embedding 向量的夹角计算。

## 1. 二维向量：图上能看到的角度

向量在几何上就是**一个带方向的箭头**。两个向量从同一个起点出发，它们之间张开的角度就是夹角。

最简单的例子，二维平面上：

```text
        ↑ B = (0, 1)
        |
        |
        |
        +————————→ A = (1, 0)
```

A 沿 x 轴正方向，B 沿 y 轴正方向，它们之间的夹角就是 **90°**——眼睛能直接看到。

再比如：

```text
A = (1, 0)
B = (1, 1)
```

B 是 45° 斜上方的箭头，所以 A 和 B 的夹角是 **45°**，也很直观。

## 2. 用公式算这个角度

光看图不够精确，用余弦公式：

$$\cos(\theta) = \frac{A \cdot B}{|A| \times |B|}$$

各部分含义：

- **A · B**（点积/内积）—— 对应分量相乘再相加。比如 A=(1,0), B=(1,1)，则 A·B = 1×1 + 0×1 = 1
- **|A|** —— 向量 A 的长度（模），用勾股定理算。|A| = √(1²+0²) = 1，|B| = √(1²+1²) = √2
- 代入：cos(θ) = 1 / (1 × √2) = 1/√2 ≈ 0.707
- 反查：θ = 45° ✓

跟图上看到的一致。

## 3. 高维向量也有夹角（关键点）

二维、三维向量的夹角好理解，**那 1024 维的 embedding 向量呢？**

人脑想象不出 1024 维空间长啥样，但**数学上夹角的定义对任意维度都成立**——只要点积和模长能算，公式就能用。

embedding 向量比如：

```python
A = [0.21, -0.84, 0.33, ..., 0.07]   # 1024 个数
B = [0.19, -0.79, 0.41, ..., 0.05]   # 1024 个数
```

照样：

- A·B = 0.21×0.19 + (-0.84)×(-0.79) + ... + 0.07×0.05（1024 项相加）
- |A| = √(0.21² + 0.84² + ... + 0.07²)
- |B| 同理
- cos(θ) = A·B / (|A|×|B|)

算出来的 cos(θ) 就反映了这两个向量在 1024 维空间里的"夹角"。**虽然你看不见这个角，但它在数学上完全有定义、可计算。**

## 4. 为什么 RAG 里用这个？

回到 embedding 的核心思路：

> "语义相近的文本，被编码成的向量在空间里方向也接近。"

"方向接近" = 夹角小 = cos(θ) 大。所以：

| cos(θ) | θ | 含义 |
|--------|----|------|
| 1 | 0° | 完全同向 → 语义几乎一样 |
| 0.8 ~ 0.99 | 小角度 | 语义很相关 |
| 0 | 90° | 正交 → 语义无关 |
| -1 | 180° | 反向 → 语义相反 |

举例（编出来便于理解）：

- "猫是哺乳动物" vs "狗是哺乳动物" → cos(θ) ≈ 0.92（夹角小，语义近）
- "猫是哺乳动物" vs "今天天气不错" → cos(θ) ≈ 0.15（夹角大，语义远）

**RAG 检索时，就是在向量库里找跟问题向量夹角最小（cos 最大）的几个文档**，把它们当上下文喂给大模型。

## 5. 为什么用 cos(θ) 而不是直接用距离？

可能会问：为什么不直接算两个向量末端的欧几里得距离（直线距离）？

因为 embedding 关心的是**语义方向**，不是**向量长度**。比如：

- "猫" 在文档里出现一次 → 向量长度 1
- "猫猫猫猫猫" 出现五次 → 向量长度可能是 5

但语义都是"关于猫"，方向一样。用余弦相似度（cos θ）会把长度归一化掉，**只看方向不看长度**，正好契合 embedding 的诉求。这也是为什么向量数据库的默认距离度量基本都是 cosine similarity。

## 一句话总结

> 向量夹角就是"两个箭头方向上差多少"的数学度量；二维三维能画出来，高维空间画不出来但公式一样能算；夹角越小，cos 值越接近 1，在 embedding 语境下就是"语义越相似"。
