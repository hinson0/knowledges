# 跨仓开发的阶段化 Coding 工作流

## 触发问题

> 能否让 Codex 同时操作两个仓库，例如 React 前端仓库 A 和 PHP + MySQL 后端/API 仓库 B？

## 关键结论

- 不建议一上来在混杂上下文里同时改两个仓库。
- 更稳的方式是：**跨两个仓库理解需求和设计方案，但实际 coding 一个仓库一个阶段推进**。
- 跨仓分析可以一起做；跨仓编码最好分阶段做。
- 这样不是因为 Codex 不能跨仓，而是因为跨仓 coding 的验证、规则、PR、回滚和上下文边界都更复杂。

## 推荐阶段

```text
阶段 0：先读两个仓库的相关代码，理解它们怎么交互
阶段 1：先改仓库 B，比如 PHP + MySQL 后端/API
阶段 2：验证仓库 B
阶段 3：再改仓库 A，比如 React 前端
阶段 4：验证仓库 A
阶段 5：做跨仓联调说明和最终验收
```

## 为什么不要混在一轮里随意改

跨仓 coding 的主要风险：

- 两个仓库技术栈不同，验证命令不同。
- 一个仓库的改动可能依赖另一个仓库的接口契约。
- 上下文容易混乱，比如把 A 仓库的规则套到 B 仓库。
- PR、回滚、上线顺序需要分开说明。
- 如果两个仓库都在同一轮里大改，review 会很难。

## 受控的跨仓节奏

把“vibe coding”改造成受控节奏：

```text
一个总需求
两个仓库任务
一个仓库一个 worktree/branch
一次只落地一个仓库的代码
每一步都有验证和交付物
```

核心原则：

- 总需求可以跨仓理解。
- 设计方案可以跨仓制定。
- 实际代码改动按仓库分阶段落地。
- 每个仓库有独立分支 / worktree、独立验证、独立 PR 说明。
- 最后再做跨仓联调、上线顺序和回滚说明。

## 实际协作流程

用户给跨仓需求：

```text
React 项目 A 要新增一个入口，调用 PHP 项目 B 的某个接口。
```

Codex 先做跨仓分析：

```text
1. 读 A/B 两个仓库
2. 总结调用链和改动点
3. 写跨仓任务拆分
```

然后先改后端/API 仓库 B：

```text
4. 在 B 仓库新建分支，接入 harness，改 API
5. 跑 B 的验证
6. 输出 B 的 PR 说明
```

再改前端仓库 A：

```text
7. 在 A 仓库新建分支，接入 harness，改页面/调用
8. 跑 A 的验证
9. 输出 A 的 PR 说明
```

最后收口：

```text
10. 给出跨仓联调清单、上线顺序、回滚说明
```

## 允许同时操作两个仓库的前提

如果确实要让 Codex 同时操作两个仓库，至少要满足：

- 两个仓库都在当前工作区可访问。
- 两个仓库都可写。
- 用户接受把执行拆成两个清晰阶段。
- 不在两个仓库之间来回随意改。
- 每个仓库的规则、验证命令、交付物分开记录。

## 经验判断

跨仓开发的最佳边界是：

```text
跨仓分析可以一起做
跨仓编码分阶段做
```

这样既保留 Codex 对整体需求和调用链的理解，又降低上下文混乱、验证遗漏、PR 难 review、回滚顺序不清的问题。
