这里的 "persistent"（持续/常驻）说的是**进程一直活着、不会自己退出**，跟"数据持久化"（存盘那种 persistence）没关系。理解它最好的方式是和普通任务对比。

Turborepo 里的任务大致分两类：

一类是**跑完就退**的（one-shot）。比如 `turbo run build`、`lint`、`test`——它们启动后干完活、产出结果、返回一个退出码，然后进程结束。Turbo 会缓存它们的输出，下次没改动就直接命中缓存。

另一类就是 **persistent 任务**。典型代表是各种 dev server：`next dev`、`vite`、`nodemon`、watch 模式的编译器。它们启动后会**一直挂着**，持续监听文件变化、持续刷新、持续往终端打日志，直到你按 Ctrl+C 把它杀掉为止。它永远不会"跑完"。

所以你在 turbo.json 里会看到这样标记：

```json
{
  "tasks": {
    "dev": {
      "cache": false,
      "persistent": true
    }
  }
}
```

`persistent: true` 是在告诉 Turbo 三件事：

这个任务永远不会自己结束，所以**别傻等它完成**。正因为它不会结束，**别的任务不能 `dependsOn` 它**——否则那个依赖方会永远卡在等待里，Turbo 会直接报错拦住你。同时它也**没法缓存**，一个永不收尾的进程根本没有"最终产物"可存。

回到你引用的那段话：一条 `turbo run dev` 之所以能"同时拉起好几个长驻服务"，正是因为 web、api、docs 各自的 dev task 都是 persistent 的——它们会**同时、持续地**往外吐日志。普通任务串行跑完就走，日志不会打架；但多个常驻进程并发地、源源不断地刷屏，混在一个终端里就糊成一团了。TUI 给每个 persistent 任务单独开一块输出区，就是为了解决这种"多个常驻进程的日志流互相干扰"的问题。

一句话记忆：**persistent = 进程常驻、不退出的长跑任务**，和那些跑一下就结束的短任务相对。
