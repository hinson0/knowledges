# 单条操作日志「队列化」评估结论

> 日期:2026-07-05 · 仓库:cms-center(操作日志功能) · 结论:**维持同步写,不队列化**

## 背景问题

已给 `docker-compose.yml` 加了 `queue` 服务(`php artisan queue:work`)。随之两个疑问:

1. 为什么 `jobs` 表没数据?
2. 把**单条 CRUD 日志的写入**也交给队列消费,是不是能**减轻服务器压力**?

## TL;DR

- `jobs` 表空 **是正常的**:操作日志的**采集/记录是同步写库**,根本不入队;唯一入队的是「**导出**」(`GenerateOperationLogExport`),且 worker 在跑时 job 会被毫秒级消费删除。
- 单条日志队列化**不能减轻服务器压力,反而增加**:用 `database` 驱动时,把 1 次 DB 写变成约 5 次 DB 读写。
- "减轻负载" ≠ "削峰"。单条 CRUD 日志**天然没有独立尖峰**(频率跟随业务写入),不需要削峰。
- 除非有实测证据(慢查询/DB 监控指向 `operation_logs` 写入),否则队列化是拿"更高总负载 + 可靠性下降 + 运维成本"换一个不存在的收益。

---

## 一、为什么 jobs 表没数据(采集不走队列)

`OperationLogService::record()` 是**同步直接写库**:

```php
// app/Services/OperationLog/OperationLogService.php:91
OperationLog::create($attributes);   // 直接 INSERT operation_logs,不 dispatch 任何 job
```

整个 `app/Services/OperationLog/` 与 `app/Support/OperationLog/` 内**无任何 `dispatch` / `ShouldQueue`**。

- **采集路径**(observer 埋点 + `record` + `logBatch`)= 同步写 `operation_logs`,与 `jobs` 表无关。加 worker 对"记日志"零影响。
- **唯一入队** = 导出:`GenerateOperationLogExport::dispatch($export->id)`(`OperationLogExportController.php:53`),`implements ShouldQueue`。只有后台点"导出"才会往 `jobs` 表写一行。
- `QUEUE_CONNECTION=database`。worker 常驻时,job 一进表毫秒级被 reserve→处理→DELETE,所以 `select * from jobs` 几乎永远抓不到。

**判表口诀:**

| jobs 表现象 | 含义 |
|---|---|
| 空 + worker 在跑 | ✅ 正常(没人 dispatch,或已被秒消费) |
| 持续堆积 | ⚠️ worker 没起/挂了/连错库 |
| `failed_jobs` 有行 | ❌ job 执行报错,去看堆栈 |

验证 worker 是否在干活:`docker logs -f cms-center-queue-container`、`php artisan queue:failed`、看 `operation_log_exports` 状态流转。

## 二、为什么单条日志「不能」天真队列化(上下文依赖)

`record()` 深度绑定**请求作用域上下文**,而 Job 在独立 worker 进程执行,这些全没有:

| 依赖 | 来源 | Job 里的下场 |
|---|---|---|
| `auth('admin')->user()` | 当前登录态(`OperationLogService.php:56`) | null → operator 全变 System |
| `operator_id/ip/user_agent` | 中间件请求级快照(`RecordOperationContext.php:16`) | 空 → 操作人/IP 全丢 |
| `site_id`(binary16) | `resolveSiteIdBytes()` 解析当前请求站点(`:84`) | 解析不到 → 命中 `:88` fail-safe → **整条日志被丢弃** |

`OperationContext` 是 `private static array`,由中间件 `handle()` 时 `set`、`terminate()` 响应后 `flush()` 清空(`RecordOperationContext.php:33`)。

**推论:** 上下文捕获(operator/ip/site_id/时间戳)**必须留在同步阶段**,队列只能接管最后那句 `OperationLog::create()`——即**一次 INSERT**。省不掉别的。

> ⚠️ 附带坑:`site_id` 是 binary(16),塞进 job payload 会触发 JSON 序列化崩溃(历史 JsonEncodingException),必须 `bin2hex` 编码、job 里 `hex2bin` 还原;`created_at` 也必须同步捕获,否则用 job 执行时间导致时间错乱;事务中 dispatch 需 `->afterCommit()`。

## 三、DB 成本账:队列化让总压力「不减反增」

`QUEUE_CONNECTION=database`,队列缓冲区就是 MySQL 的 `jobs` 表:

| 方案 | 打在 MySQL 上的操作 | 次数 |
|---|---|---|
| 现在(同步) | `operation_logs` INSERT | **1 次** |
| 队列消费 | `jobs` INSERT → worker `SELECT..FOR UPDATE` → `UPDATE`(reserve) → `operation_logs` INSERT → `jobs` DELETE | **约 5 次** |

为省 1 次日志 INSERT,给同一个 MySQL **增加约 5 次读写** + 序列化/反序列化 + worker 常驻内存 + 调度 CPU。**服务器干的活变多,只是换了进程。**

外加可靠性下降:同步失败只丢单条 + `Log::warning`;队列失败进 `failed_jobs`,**日志可能永久丢**,还多一份监控。

---

## 四、核心误区:减轻负载 ≠ 削峰

队列的价值不是"少干活",而是**削峰填谷**——把瞬时高峰摊到时间轴上,避免峰值打垮系统。

> 比喻:队列不是"少洗几个碗",而是"午高峰先把碗堆水槽,下午人少慢慢洗"。碗总数没变(搬来搬去甚至更多),但避免了高峰手忙脚乱。

所以真正的问题是:**单条 CRUD 日志有需要削的"峰"吗?没有。**

1. **频率跟随业务**:一次操作 = 一次业务写 + 一次日志写。日志 INSERT 频率天然 ≈ 业务写入频率。若单条日志能压垮 DB,业务写入早就先垮了——日志不是瓶颈。
2. **批量已聚合**:真正的尖峰是批量操作,但 `logBatch` 已把一批聚合成 **1 条**日志(`OperationLogService.php:20-35`),尖峰早被削平。
3. **缓冲在同库**:`database` 驱动的缓冲(`jobs`)和目标(`operation_logs`)打在**同一个 MySQL**,峰值时两边抢同一个库的写 IO,**削峰效果基本抵消**。真要削峰得换 Redis 队列,把缓冲移出 MySQL。

---

## 五、若哪天日志写入「真」成瓶颈(有实测再做)

正确方向**不是**"每条 dispatch 一个 job",而是:

1. **批量落库**:`afterResponse` 里攒多条日志一次 `insert([...多行...])`,减少 INSERT **次数**(这才真减压)。
2. **Redis 队列 + 批量 flush worker**:日志先进 Redis(内存,不碰 MySQL),worker 攒够一批再一次性写库——真正把写压力移出 MySQL + 削峰。
3. **控制表体积**:已有的 **90 天清理方案**。表小 INSERT 才快,比队列化实在。

均属 YAGNI 范畴,有实测瓶颈才做。

---

## 六、结论对照表

| 诉求 | 队列(database)能否满足 |
|---|---|
| 减轻服务器**总负载** | ❌ 总活儿变多(1 次 DB 操作 → 约 5 次) |
| 降低**接口响应延迟** | ✅ 能,但 `afterResponse` 更轻,不必上队列 |
| **削峰**防高峰压垮 | ⚠️ 单条日志无独立尖峰;缓冲在同库,削峰打折 |

**最终建议:维持现状同步写。** 单条日志 INSERT 是个位数毫秒,不是瓶颈。想减压优先"控制表体积(90 天清理)";想降延迟用 `afterResponse`;要削峰才考虑 Redis 队列 + 批量落库,且需实测证据支撑。
