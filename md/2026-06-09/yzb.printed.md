# 第一周计划和目标

第一周以“通过标准 Harness 工程和标准 AI 开发流程，快速完成一个独立需求的最小版本开发”为核心目标。员工需要在产品配合下，围绕真实业务需求完成需求理解、任务拆解、方案确认、开发实现、验证反馈和复盘沉淀，重点不是追求一次性做完整，而是通过最小可用版本快速跑通 AI Coding 研发闭环。

通过第一周实践，员工需要初步建立对标准 Harness 工程、AI 开发流程、团队协作规范和需求交付节奏的理解；在实际开发过程中积累成功经验和失败案例，形成可复用的问题清单、优化建议和操作方法，为后续第 2 个到第 N 个需求的持续交付、快速响应和快速优化打基础。

## 第一周核心目标

| 目标 | 验收标准 |
|---|---|
| 明确独立需求范围 | 与产品完成需求对齐，明确需求目标、最小版本范围、验收标准、非范围和关键风险 |
| 使用标准 Harness 工程开展开发 | 按标准 Harness 工程组织上下文、代码入口、任务拆解、开发记录和验证材料 |
| 跑通标准 AI 开发流程 | 按需求理解、任务拆解、方案确认、代码生成 / 修改、测试验证、问题复盘的流程完成一次闭环 |
| 完成最小版本开发 | 交付一个可运行、可演示、可验证的独立需求最小版本，不要求一次性覆盖全部扩展能力 |
| 快速响应与快速优化 | 根据产品、导师或测试反馈，至少完成 1 轮问题修正、方案优化或体验调整 |
| 沉淀成功 / 失败经验 | 记录 AI Coding 使用过程中的有效 Prompt、上下文组织方式、失败输出、踩坑问题和改进动作 |
| 形成后续需求优化方向 | 输出下一个需求可复用的流程建议、检查清单或注意事项，为第 2 个到第 N 个需求提效 |

## 第一周重点要求

1. 第一周不以“大而全”的复杂交付为目标，优先通过一个独立需求的最小版本跑通标准研发闭环；
2. 需求开始前必须与产品明确目标、范围、验收标准和非范围，避免边做边发散；
3. 开发过程中必须使用标准 Harness 工程和标准 AI 开发流程，不能只依赖临时 Prompt 或零散对话完成开发；
4. AI 生成的代码、方案和结论必须结合源码、运行结果、测试验证和导师反馈进行确认，不能直接作为最终结果；
5. 对开发过程中的成功经验、失败案例、返工原因和优化动作要及时记录，形成可复用沉淀；
6. 遇到需求不清、上下文不足、AI 输出不稳定、验证不充分等问题，需要快速反馈、快速修正、快速优化；
7. 周末前需要完成一次简短复盘，说明本周独立需求的完成情况、主要问题、有效方法和后续需求优化方向。

# 重点工作要求

## 1. AI Coding 全周期实践要求

AI Coding 实践、总结和分享需要贯穿整个试用期，每个月都要有可见产出。重点包括：

1. 需求阶段：使用 AI 辅助澄清需求、拆解任务、识别边界和风险；
2. 设计阶段：使用 AI 辅助生成技术方案、接口设计、数据流说明和影响面分析；
3. 开发阶段：使用 AI 辅助代码生成、重构、问题定位和单元测试补充；
4. 验证阶段：使用 AI 辅助生成测试用例、检查异常路径、补充日志和回滚方案；
5. 复盘阶段：沉淀有效 Prompt、上下文模板、失败案例、检查清单和最佳实践。

AI 只能作为辅助工具，所有代码、方案和结论必须由本人负责验证，不能直接依赖 AI 输出作为最终结果。

## 2. Harness 改造要求

Harness 改造需要结合真实老 PHP 项目的实际场景推进，避免只停留在方案或 Demo。交付内容至少包括：

1. 明确一个老 PHP 项目的核心改造场景，如代码定位、需求理解、上下文组织、测试辅助、文档生成或常见问题排查；
2. 完成可运行、可演示的 Harness 改造结果，并能说明改造前后的效率或质量变化；
3. 输出配套说明文档，包括使用方式、适用场景、边界限制、注意事项和后续优化方向；
4. 将改造结果用于至少 1 个真实需求或真实问题处理；
5. 对落地过程进行复盘，记录问题、收益、风险和下一步计划。

## 3. 过程管理要求

1. 每月进行一次阶段评估，评估内容包括需求交付、Harness 进展、AI Coding 沉淀、培养赋能、协作反馈和问题改进；
2. 每月至少输出 1 份阶段性文档或复盘材料；
3. 重要需求需要保留研发设计、测试验证、上线结果和问题复盘记录；
4. 对返工、延期、线上问题或协作问题，需要明确原因分析和改进动作；
5. 第 6 个月进行综合转正评估，重点看实际产出、可复用沉淀、成长变化和后续成长潜力。

# 培养与赋能机制

## 1. 导师辅导机制

试用期内由导师对员工进行持续辅导，重点不是简单分配任务，而是帮助员工理解项目、掌握方法、发现问题并持续改进。

1. 第 1 个月重点辅导项目结构、业务流程、研发规范、上线流程和 Harness 改造思路；
2. 第 2-3 个月重点辅导需求拆解、AI Coding 使用方法、Prompt 编写、上下文组织和测试验证；
3. 第 4-5 个月重点辅导复杂需求处理、影响面分析、工程质量控制、跨团队协作和方法沉淀；
4. 第 6 个月重点辅导试用期成果总结、能力复盘、短板改进和后续成长规划。

## 2. 阶段反馈机制

每月至少进行一次正式阶段反馈，反馈内容包括：

1. 本月需求完成情况、交付质量和协作表现；
2. Harness 改造进展和实际应用效果；
3. AI Coding 使用中的有效方法、典型问题和风险点；
4. 文档沉淀、复盘质量和工程化意识；
5. 下个月需要重点提升的能力和具体行动。

反馈应尽量具体，避免只给笼统评价。对于明显问题，需要给出可执行的改进建议和跟进时间。

## 3. 专项学习与实战训练

围绕岗位要求，为员工安排必要的专项学习和实战训练，帮助员工快速补齐能力短板。

1. 老 PHP 项目专题：项目结构、历史背景、核心业务链路、常见问题和风险点；
2. Harness 专题：核心概念、落地方式、适用场景、边界限制和典型案例；
3. AI Coding 专题：上下文组织、Prompt 编写、代码生成、测试补充、结果验证和失败案例复盘；
4. 工程质量专题：日志、异常处理、特性开关、回滚路径、测试用例和代码评审标准；
5. 协作流程专题：需求评审、开发联调、测试验收、预发验证、上线发布和复盘机制。

## 4. 结对实践与案例复盘

对于关键场景，导师或资深同事可以通过结对实践和案例复盘帮助员工提升实战能力。

1. 首月可安排 1-2 次老项目代码走查，帮助员工快速建立项目上下文；
2. 关键需求可安排方案评审或结对拆解，帮助员工提升需求理解和任务拆分能力；
3. AI 生成代码需要结合代码评审进行验证，帮助员工建立“不盲信 AI 输出”的工程习惯；
4. 对返工、延期、线上问题等情况，进行专项复盘，帮助员工形成问题分析和改进能力。

## 5. 成长记录与能力沉淀

试用期内需要保留员工成长过程中的关键记录，作为转正评估的重要依据。

1. 每月记录一次阶段成长情况，包括已提升能力、暴露问题和改进动作；
2. 重要需求保留研发设计、AI Coding 使用过程、测试验证和上线结果；
3. Harness 改造需要保留方案、落地结果、使用说明和复盘记录；
4. AI Coding 实践需要保留有效 Prompt、上下文模板、失败案例和检查清单；
5. 第 6 个月形成完整的试用期成长总结，明确转正后的能力提升方向。

# 转正标准

| 返工控制 | 明显返工 ≤ 2 个，且有复盘和改进措施 | 因方案不清、验证不足导致多次返工 |
