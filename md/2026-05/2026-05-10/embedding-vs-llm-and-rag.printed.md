# Embedding / 余弦相似度 / RAG 概念三件套

> 来源：week2/day5_workspace/0510_2/1715.md + 1725.md。day5 长期记忆所需的 RAG 基础概念体系。

## 1. embedding 是什么？

**一句话**：把一段文本 → 一串数字（向量），让计算机能"算两段文本相似不相似"。

```text
"用户硬件 Mac M2"        → bge-m3 → [0.13, -0.42, 0.81, ..., 0.05]   ← 1024 个数字
"我电脑啥配置"            → bge-m3 → [0.15, -0.39, 0.76, ..., 0.08]   ← 1024 个数字
"今天天气不错"            → bge-m3 → [-0.22, 0.55, -0.11, ..., 0.91]  ← 1024 个数字
```

**核心魔法**：语义接近的文本，向量也接近（欧式距离近 / 余弦角度小）。

```text
"我电脑啥配置" 和 "用户硬件 Mac M2"  → 余弦相似度 0.57   ← 高（语义相关）
"我电脑啥配置" 和 "今天天气不错"      → 余弦相似度 0.51   ← 较低
```

### 为什么 day5 需要它？

memory 库里存了 100 条历史摘要，新 thread 用户问"我电脑配置是啥"，怎么知道该召回哪条？

- ❌ 关键词匹配（SQL `LIKE '%电脑%'`）：用户问"硬件配置"就漏了，问"machine spec"也漏
- ❌ 给 LLM 看 100 条让它选（贵 + 慢 + context 装不下）
- ✅ **算 embedding 余弦相似度**：毫秒级，无关键词依赖，跨语言

这就是 RAG 的底层引擎。

### embedding 模型的本质

字符串层面 0% 重合的字符，可以在向量层面非常相近：

```text
计算机不会"算两个字符串有多像":
  "猫" 和 "狗"     是相似还是不相似?字符层面 0% 重合
  "猫" 和 "Cat"   是相似还是不相似?字符层面 0% 重合
  "猫科动物"       和 "猫" 跟 "猫的房子" 哪个更像?

embedding 模型把这些字符串"翻译"成向量后,变成数学题:
  "猫"          → [0.81, 0.12, -0.33, ..., 0.05]
  "狗"          → [0.79, 0.18, -0.29, ..., 0.07]    ← 跟猫很近
  "Cat"         → [0.80, 0.13, -0.31, ..., 0.06]    ← 也跟猫很近(跨语言!)
  "今天下雨"     → [-0.42, 0.91, 0.05, ..., -0.66]   ← 跟猫很远
```

**关键魔法**：embedding 模型在训练时被反复看了上亿对"语义相似"的句子，学会"意思接近的句子，向量也接近"。

```text
embedding 模型 = "语义指纹生成器"
                  同样的意思 → 同样的指纹
                  不同的意思 → 不同的指纹
```

## 2. embedding 模型 vs LLM（deepseek v4）：本质不同

| 维度 | LLM（deepseek v4 flash/pro） | embedding 模型（bge-m3） |
|------|------------------------------|--------------------------|
| 输入 | 文本（prompt + 历史） | 文本（单段） |
| 输出 | **文本**（自然语言生成） | **数字向量**（1024 维 float） |
| 任务 | "会说话" — 生成、对话、推理、调工具 | "会标坐标" — 把文本投到高维空间 |
| 模型大小 | 几百亿参数 | bge-m3 约 5.7 亿参数（小 100 倍）|
| 单次调用成本 | 贵（input + output token 都收费）| 便宜（只算 input token）|
| 单次延迟 | 1-5 秒 | 100-300ms |
| 维度概念 | "上下文长度"（1M token）| "向量维度"（1024 维）|

**形象类比**：

```text
LLM 像图书馆的"导读员":
  你问问题,它读理解,组织语言,讲给你听。

Embedding 模型像图书馆的"索书号生成器":
  给它一段文本,它告诉你"这本书该放在书架哪个坐标"。
  作用是让你能快速找到"附近的书"(语义相近的内容)。
```

## 3. 能不能用 deepseek v4 替代 embedding？

**理论上能（用 prompt 让 LLM 输出向量数字），工程上完全错误**，原因有 4 个：

### ❌ 原因 1：成本天差地别

假设 memory 库有 1000 条摘要，每次 search 要算 1 次 query embedding + 跟 1000 条对比：

| 方案 | 单次 search 成本 |
|------|----------------|
| bge-m3（硅基流动）| ≈ 0.0002 元（只 embed 1 次 query，算余弦不调 API）|
| deepseek v4 flash 模拟 | ≈ 几元（每条都要 LLM 调用）|
| deepseek v4 pro 模拟 | ≈ 几十元 |

**RAG 一天可能 search 几千次，成本量级就是几百倍**。

### ❌ 原因 2：LLM 不输出"稳定可比"的向量

给 LLM prompt："把这句话变成 1024 个 float"。它每次输出的数字会变（LLM 本质是采样），**两次同一句话的"向量"不一样，余弦也不稳定**。embedding 模型是确定性的：同一句话进去，永远是同一个向量。

### ❌ 原因 3：速度慢 10-50 倍

embedding 模型架构上专门为"快速吐向量"优化（只跑 encoder，不做 decoder 生成）。LLM 必须生成 token by token，1024 个数字要生成几千个 token，**100ms vs 5 秒的差距**。

### ❌ 原因 4：训练目标不一样

bge-m3 训练时被反复"折磨"：让它学会"语义相近的句子向量距离要近"。LLM 训练目标是"下一个 token 是什么"，**根本没被训练过"两句话有多像"**。强行拿 LLM 内部 hidden state 当向量（OpenAI 早期就这么干过）效果远不如专用 embedding 模型。

## 4. 余弦相似度：为什么余弦能算相似度？

**先看高中数学的余弦**：

```text
两个向量 A 和 B 之间夹角 θ
cos(θ) = A · B / (|A| × |B|)

θ = 0°    → cos = 1.0    向量同方向
θ = 90°   → cos = 0.0    向量垂直
θ = 180°  → cos = -1.0   向量反方向
```

**embedding 用余弦**：把每个文本看成高维空间（1024 维）里的一支**箭**（向量），箭头指向某个方向。

```text
   ↑
   │   ← "猫"    箭头方向 A
   ↗   ← "狗"    箭头方向 B,夹角小(15°),cos ≈ 0.97
   │
   ←     ← "今天下雨"  跟"猫"夹角大(120°),cos ≈ -0.5
```

**为什么用余弦，不用欧式距离？**

| 距离方式 | 算的是什么 | 缺点 |
|---------|----------|------|
| 欧式距离 `\|A - B\|` | 两点的**直线距离** | 受向量"长度"影响：句子长的向量长，距离自然就大 |
| 余弦相似度 `cos θ` | 两个箭头的**方向夹角** | **只看方向，不看长度** —— 短句和长句也能公平比 |

文本长度不一，但你只关心"意思方向是否一致" → 余弦更合适。

**实际数字感觉**：

```python
cos = 1.0     # 完全一致(同句子)
cos = 0.7-0.9 # 高度相关("猫" vs "猫科动物")
cos = 0.4-0.6 # 弱相关 / 同语言基线噪音(中文里两句无关话也常 0.5)
cos = 0.0     # 无关
cos < 0       # 反义(实际 embedding 很少出现)
```

代码里就一行：

```python
cos = np.dot(A, B) / (np.linalg.norm(A) * np.linalg.norm(B))
```

`np.dot(A, B)` 是点积（对应位相乘求和），分母是两个向量的"长度"，合起来就是数学定义的 cos θ。

## 5. RAG 是什么？

**RAG** = **R**etrieval-**A**ugmented **G**eneration = **检索增强生成**。

### 为什么要这玩意？LLM 不是已经会回答问题吗？

LLM 有 3 个硬伤：

1. **知识停留在训练时**（deepseek v4 知识截止 2025-04，问"昨天的新闻"它瞎编）
2. **不知道你私有的东西**（你公司代码 / 你 memory 库 / 你的 PDF）
3. **context 装不下**（1M 上下文也不能把 10 万行代码全塞进去）

### RAG 的解法（一句话）

**回答前先去外部库捞相关资料，塞进 prompt 再让 LLM 答**。

```text
传统 LLM:
  用户问 → LLM 凭训练知识答 → 回答

RAG:
  用户问 → ① 把问题 embedding → ② 去向量库找最像的 N 条资料
        → ③ 把资料 + 用户问题一起塞给 LLM → ④ LLM 基于资料答 → 回答
```

**day5 就是 RAG 最小形态**（只是"资料库"是 memory 摘要，不是文档）：

```text
用户在 thread2 问: "我电脑啥配置?"
  ↓
① embedding("我电脑啥配置") = [0.15, -0.39, ...]
  ↓
② 去 SQLite 里跟所有 memory 算余弦,排序拿 top-3
  ↓
③ 把 top-3 拼成 SystemMessage:"## 已知事实\n- 用户硬件 Mac M2..."
  ↓
④ LLM 看到系统消息后回答:"你电脑是 Mac M2,Python 3.12,主要写 FastAPI"
```

**week4 你做的"代码 RAG"是同一套流程**，只是资料库从摘要换成代码 chunks：

```text
用户问"修一下 user.login 的 bug"
  ↓
① embedding("user.login 函数")
  ↓
② 去向量库找最像的 5 段代码
  ↓
③ 把代码 + 问题塞给 LLM
  ↓
④ LLM 看代码后给修复方案
```

**RAG 三个核心组件**（day5 你都接触了）：

| 组件 | day5 用什么 | week4 用什么 |
|------|------------|-------------|
| 资料库 | SQLite memories 表 | Chroma 向量库 |
| 检索引擎 | embedding（bge-m3）+ 余弦排序 | embedding + BM25 + reranker |
| 生成器 | deepseek v4 flash | 同 |

## 6. day5 你的 stack 长啥样？

**两个模型分工合作**：

```text
┌─ deepseek v4 flash(LLM)─────────────┐
│  • 跟用户对话(call_llm 节点)         │
│  • 调工具(read_file / 等)            │
│  • 写摘要(summarize 节点 — 你要写的)│
└──────────────────────────────────────┘
         ↓ 摘要文本(短句)
┌─ bge-m3(embedding 模型)─────────────┐
│  • 把摘要变向量存 SQLite             │
│  • 把 query 变向量算余弦召回         │
└──────────────────────────────────────┘
```

**deepseek 负责"动嘴"，bge-m3 负责"标坐标"**。两个都不可少。

## 坑 / Why · 整体认知

- **RAG 的本质是"让 LLM 开卷考试"**。LLM 闭卷答（瞎编）→ 给它递参考资料 → 它照着答（准）。这个朴素直觉就是 RAG 的全部价值，后面所有"chunk 切分策略"、"hybrid search"、"reranker"都只是在解决"怎么递更准的资料"这一个问题。
- **embedding + 余弦不是 RAG 的唯一检索方式**。还有 BM25（关键词匹配，2009 年的老技术）、reranker（精排，小模型给 query-doc 打分）。bge-m3 流行的关键是：它**一个模型同时支持 dense embedding + sparse（类似 BM25）+ multi-vector（类似 ColBERT）三种检索**。week4 hybrid search 就是把这三条腿全用上。
- **day5 你在做 RAG 的最小完整形态**：存（put）+ 检（search）+ 注入（inject）+ 生成（LLM）。week4 升级版只是把"存"做精（代码 chunking）、"检"做强（reranker）。**架构骨架这周就定型了，下周你只是在原地优化**。所以 day5 跑通 = 你已经会 RAG 了，差的只是工程精度。
- **week4 RAG 周你会接触第三类模型：Reranker（`bge-reranker-v2-m3`）**。它的任务又不一样：input 是 (query, doc) 对，output 是单个相关性 score。"先 embedding 粗排 top-20，再 reranker 精排 top-3"是 RAG 工程的黄金组合——**三类模型各司其职**：LLM（生成）、embedding（投影）、reranker（打分）。day5 只用前两类，week4 才补全。
- **bge-m3 这个名字暗藏 3 个能力**：**M**ulti-functionality（同时支持 dense / sparse / colbert 三种向量模式）、**M**ulti-linguality（100+ 语言）、**M**ulti-granularity（短句到 8192 token 长文都行）。day5 你只用了 dense 模式 1024 维——它还有另外两条腿（sparse 跟 BM25 类似，colbert 多向量精度更高），week4 你会撞上"hybrid search"用全这三种。
- **embedding 在 AI 工程里被叫做"语义指纹"**：同一个意思的不同说法、不同语言、不同表达，指纹应该相似。**day5 SCORE_THRESHOLD 调优，本质是在调"指纹有多相似才算同一个意思"**——这个判断没有标准答案，只能跑数据看分布。这就是 RAG 工程"经验主义"的根源，也是简历能写出"通过 30 条 query 实验把准确率从 X 提到 Y"的金矿来源。

## 一图总结（三个概念的关系）

```text
        文本字符串
            ↓
     ┌──────────────┐
     │ embedding 模型 │   ← 1. 把文本变向量(指纹生成器)
     └──────────────┘
            ↓
       1024 维向量
            ↓
     ┌──────────────┐
     │  余弦相似度    │   ← 2. 算两个向量的"方向夹角",越接近 1 越像
     └──────────────┘
            ↓
       排序拿 top-N
            ↓
     ┌──────────────┐
     │  RAG 流程     │   ← 3. 把 top-N 资料塞给 LLM,让它"开卷答题"
     └──────────────┘
            ↓
       LLM 给出有据回答
```
