你说得对，你没有告诉我分支。我当时是自己推断的，具体是：

```text
cms-center
本地分支: feature/social-media-share
来源: origin/dev

ce-site-backend
本地分支: feature/social-media-share
来源: origin/dev

ce-designer
本地分支: feature/social-media-share
来源: origin/feature/element-share
```

我为什么这么选：一开始从默认分支看不到完整的 `social-share` 实现，所以我搜索了 remote branches。结果是：

- `cms-center` 的社媒分享接口在 `origin/dev` / `origin/test` 里能看到；
- `ce-site-backend` 的管理页实现也在 `origin/dev` / `origin/test` 里能看到；
- `ce-designer` 相关组件代码不在 `dev`，而是在 `origin/feature/element-share`。

所以我创建了同名本地 feature 分支：

```bash
git switch -C feature/social-media-share origin/dev
git switch -C feature/social-media-share origin/feature/element-share
```

但这个决策我应该在动代码前明确告诉你，至少应该说：“我发现实现不在默认分支，准备从这些分支建 worktree，可以吗？” 尤其 `ce-designer` 用的是 `feature/element-share`，这是一个更强的假设，应该确认。

后续我建议把 CE 流程规则改成：

- 用户给 repo URL 但没给分支时，先检查默认分支是否有目标实现；
- 如果需要切到非默认分支，必须先列出来并确认；
- 不要静默从 `dev`、`test` 或 feature branch 推断。

当前这次 debug 的实际基线就是上面三个来源分支。
