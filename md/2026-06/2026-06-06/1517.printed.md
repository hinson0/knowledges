AGENTS.md 可以理解成 **“给 Coding Agent 看的 README”**：README 给人看，AGENTS.md 给 Codex / Claude Code / Cursor Agent 这类工具看。它不要写成愿景文档，要写成**可执行的工程约束**。

OpenAI 官方 Codex 文档也建议可以放两层：全局 `~/.codex/AGENTS.md` 放个人长期偏好，仓库根目录 `AGENTS.md` 放项目规则；项目规则会用于约束该仓库里的 Codex 工作。([OpenAI 开发者][1]) 另外 agents.md 社区也把它定义成一种面向 coding agent 的简单开放格式。([agents.md][2])

## 一般结构

我建议你按这个顺序写：

```md
# AGENTS.md

## 1. 项目概览

一句话说明这个仓库是干什么的。

## 2. 技术栈

列清楚前端、后端、包管理器、数据库、测试框架。

## 3. 常用命令

安装、启动、构建、测试、lint、typecheck 命令。

## 4. 工作规则

Agent 工作时必须遵守的流程，例如不能直接改 master，必须开 worktree，改代码前先读相关文件。

## 5. 代码风格

命名、目录、组件、API、错误处理、日志、类型约束。

## 6. 测试规则

什么时候必须补测试，怎么跑测试，最低验证标准是什么。

## 7. Git / PR 规则

分支命名、commit 规则、PR 描述要求、禁止事项。

## 8. 安全 / 禁止操作

不能删数据、不能改生产配置、不能引入大依赖、不能重写历史等。

## 9. 任务完成检查清单

Agent 每次结束前必须自检什么。
```

## 一个比较实用的模板

你可以直接从这个版本开始：

````md
# AGENTS.md

## 项目概览

这个仓库是一个 Harness 工程基础设施项目，用来标准化多个项目中的 AI coding agent 工作流。

目标是为 Codex、Claude Code、Cursor 等工具提供可复用的规则、skills、模板和工作流。

## 技术栈

- 包管理器：pnpm
- 运行时：Node.js
- 前端：React / Next.js（如适用）
- 后端：Node.js / API routes（如适用）
- 数据库：如有使用则写明
- 测试：写明实际使用的测试框架

## 常用命令

在这个仓库工作时使用这些命令：

```bash
pnpm install
pnpm dev
pnpm lint
pnpm typecheck
pnpm test
pnpm build
```

如果某个命令不存在，先检查 `package.json`，不要直接假设命令名称。

## 工作规则

- 不要直接在 `master` 或 `main` 上工作。
- 修改前先检查相关文件，理解现有结构。
- 优先做小而聚焦的修改，不要大改重写。
- 完成任务时不要重构无关代码。
- 未经明确批准，不要引入新的生产依赖。
- 除非任务明确要求改变行为，否则保持现有行为不变。
- 不确定时，先明确说明假设再继续。

## Worktree 规则

- 非简单的功能工作使用单独的 git worktree。
- 分支名称应遵循这个模式：

```bash
feature/<short-task-name>
fix/<short-bug-name>
chore/<short-maintenance-name>
docs/<short-doc-name>
```

- 不要直接在 `master` 或 `main` 上创建功能改动。
- 开始前检查当前分支和 git status。

## 代码风格

- 优先写清晰、明确的代码，不要追求巧妙抽象。
- 保持函数小而内聚。
- 使用有意义的命名。
- 避免隐藏的全局状态。
- 避免魔法常量；当常量有业务含义时给它命名。
- 公共工具的行为不明显时，应补充注释或文档。
- 显式处理错误。
- 日志要可用于定位问题，避免嘈杂的 debug log。

## 测试规则

- 修改代码后运行相关测试。
- 在认为任务完成前运行 `pnpm lint` 和 `pnpm typecheck`。
- 以下情况需要新增或更新测试：
  - 修复 bug
  - 修改业务逻辑
  - 添加新功能
  - 修改公开 API

- 如果测试无法运行，说明原因，并列出手动检查过的内容。

## Git 规则

- 保持 commit 聚焦。
- 除非项目明确要求，否则不要提交生成文件。
- 除非依赖发生变化，否则不要修改 lockfile。
- 除非明确要求，否则不要重写 git history。
- 不要删除用户工作或 untracked files。

## 文档规则

- 当行为、命令、设置或工作流发生变化时，同步更新文档。
- 不要在这个文件里重复大段需求。
- 当细节对 AGENTS.md 来说太长时，链接到专门的文档。

## 禁止操作

除非明确要求，否则不要做这些事：

- 删除与任务无关的文件。
- 修改生产环境配置。
- 添加大型框架或依赖。
- 为了让构建通过而禁用测试、lint 规则或类型检查。
- 未经讨论就替换现有架构。
- 触碰 secrets、credentials、tokens 或 `.env` 值。
- 运行破坏性的数据库或文件系统命令。

## 任务完成检查清单

结束前确认：

- 修改符合用户请求。
- 实现保持最小且聚焦。
- 已运行相关测试、lint 和类型检查，或已记录无法运行的原因。
- 没有修改无关文件。
- 没有引入 secrets 或仅本地有效的路径。
- 最终回复总结：
  - 修改了什么
  - 如何验证
  - 剩余风险或后续工作

````

## 你做 Harness 的话，重点要加这几类规则

普通项目的 AGENTS.md 只写“怎么跑项目”。但你这个 Harness 更重要的是**制度化 agent 行为**，所以建议额外加：

```md
## Harness 原则

- 这个仓库定义可复用的 AI coding agent 工作流。
- 优先编写可跨项目复用的规则。
- 区分通用规则和项目专属规则。
- 不要硬编码只适用于单个仓库的假设。
- 新增规则时包含：
  - 它要避免的问题
  - 期望行为
  - 如有帮助，补充示例
```

再加一个：

```md
## Agent 指令设计规则

- 指令应该具体且可测试。
- 避免使用“写好代码”这类模糊规则。
- 优先使用“修改 TypeScript 文件后运行 `pnpm typecheck`”这类规则。
- 指令要足够短，让 agent 每个会话都能读完。
- 把长解释移到单独文档中，并在这里链接过去。
```

## 最佳实践：不要写太长

AGENTS.md 最怕变成“第二个知识库”。我的建议是：

```txt
AGENTS.md = 高频、强约束、每次都要看的规则
docs/*.md = 低频、长解释、背景知识
skills/* = 可复用的专项流程
scripts/* = 可以自动化执行的检查
```

比如：

```md
## 参考资料

- 工程工作流：`docs/engineering-workflow.md`
- Worktree 指南：`docs/git-worktree.md`
- Codex 使用指南：`docs/codex-best-practices.md`
- Skill 编写指南：`docs/skills.md`
```

## 一个判断标准

写 AGENTS.md 时，每一条都问自己一句：

> 这条规则能不能改变 agent 的实际行为？

如果不能，就删掉。

比如：

```md
- 编写高质量代码。
```

这句没用。

改成：

```md
- 优先编写职责单一的小函数。
- 至少出现两个真实调用点后，再引入共享抽象。
- 修改 TypeScript 代码后运行 `pnpm typecheck`。
```

这就有用了。

## 推荐你的落地方式

你可以先做三层：

```txt
~/.codex/AGENTS.md
```

放你的个人习惯，比如：

```md
- 讨论架构时优先使用中文解释。
- 优先使用 pnpm。
- 添加生产依赖前先询问。
- 始终总结验证步骤。
```

项目根目录：

```txt
AGENTS.md
```

放这个仓库的硬规则，比如技术栈、命令、测试、分支、worktree。

子目录：

```txt
frontend/AGENTS.md
backend/AGENTS.md
infra/AGENTS.md
```

放局部规则，比如前端组件规范、后端 API 规范、infra 脚本规范。

一句话总结：

> AGENTS.md 不要写“项目介绍文”，要写“Agent 操作手册”；不要追求完整，追求每条规则都能约束一次真实编码行为。
