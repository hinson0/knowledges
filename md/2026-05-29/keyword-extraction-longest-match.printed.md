# 关键字提取:词典 + 最长匹配(避免臆造细节)

## Trigger Question

> 这样的问题,我输入的东西没有双人沙发,但是为什么关键字里面有双人沙发?

## Key Takeaways

- 提取器 **不回显你打的字**,而是把文字撞一个 **词典**,每条词典有「触发词列表 + 一个固定展示值」;命中触发词就吐出那个固定值。
- 原 bug:沙发词条 `{ terms:["双人沙发","沙发"], value:"沙发(双人)" }` —— 只打「沙发」也命中,显示成 demo 那版「沙发(双人)」,**臆造了你没说的「双人」**。
- 根因:展示值里写死了 demo 专用的具体程度(demo 原句是「双人沙发」),却被一个更泛的触发词「沙发」共用。
- 修法:**按命中的最长词取值** —— 拆成两条(`双人沙发→沙发(双人)`、`沙发→沙发`),并在提取时「若某命中词被另一更长命中词包含,丢弃较短的」。
- 效果:打「沙发」→「沙发」;打「双人沙发」/demo →「沙发(双人)」(且只一个主体标签)。规则通用,只要短词是长词子串就让更具体的胜出;跨维度(如「实木沙发」→ 材质实木 + 主体沙发)互不影响。

## Code Example

```ts
// 拆分词条:具体的与泛的分开,各带自己的值
{ terms: ["双人沙发"], dim: "主体", value: "沙发(双人)" },
{ terms: ["沙发"],     dim: "主体", value: "沙发" },

// 最长匹配:被「更长且包含自己」的命中词覆盖的较短词,丢弃
const winners = matched.filter(
  (m) => !matched.some(
    (o) => o.term.length > m.term.length &&
           o.term.toLowerCase().includes(m.term.toLowerCase()),
  ),
);
```

## Pitfall / Why

- 词典型提取的通病:把「规范展示值」和「触发词的具体程度」耦合在一起,泛触发词会带出过度具体的值。最长匹配是最小且通用的解。
- 英文触发词记得 `toLowerCase` 双侧统一(CJK 不受影响)。
- 单字触发词(如「床」)易误命中(起床/床单),属已知小坑;此词典为 demo 规模,未深究。
