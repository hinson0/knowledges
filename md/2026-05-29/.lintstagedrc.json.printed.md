`.lintstagedrc.json` 是 **lint-staged 这个工具的配置文件**。拆开讲:

## 名字怎么来的

- **lint-staged** = "对 **staged(已暂存)** 的文件运行命令"。它本来是为"提交前只 lint 改动文件"设计的,但运行什么命令完全由你定——我们让它运行 Prettier。
- **rc** = "runtime configuration"(运行时配置),是 Unix 老传统(`.bashrc`、`.eslintrc`、`.npmrc` 都是这个 `rc`)。所以 `.lintstagedrc.json` = "lint-staged 的配置,JSON 格式"。

## 它里面写了什么

```json
{
  "*": "prettier --write --ignore-unknown"
}
```

读法:**键是文件匹配模式,值是要对匹配到的暂存文件运行的命令**。

- `"*"` —— 匹配所有文件。
- `"prettier --write --ignore-unknown"` —— 对它们跑 Prettier 改写;`--ignore-unknown` 让 Prettier **跳过它不认识的文件**(比如图片、二进制),而不是报错。

`★ Insight ─────────────────────────────────────`
**关键机制:lint-staged 会把"匹配到的暂存文件名"自动拼接到命令末尾。** 所以实际执行的是:
```
prettier --write --ignore-unknown 你这次git add的文件1 文件2 ...
```
它**只处理你本次 `git add` 进暂存区的文件**,不是整个仓库。这正是它和 `pnpm format`(格式化全仓库)的区别——同一个 Prettier,作用范围不同。
`─────────────────────────────────────────────────`

## 它在整个流程里的位置

它本身不会自动跑,是被 **`.githooks/pre-commit`** 调用的。完整链路:

```
你 git commit
  └─ pre-commit hook 触发
       └─ 调用 lint-staged
            └─ 读 .lintstagedrc.json
                 └─ 对你暂存的文件跑 prettier --write
                      └─ 把格式化后的文件重新加回暂存区 → 进入这次提交
```

**一个具体例子**:你改了 `ChatMain.tsx`(格式很乱)+ `route.ts`,`git add` 这两个后 `git commit`。lint-staged 只对这 2 个文件跑 Prettier,把它们格式化好再提交——其余 100 多个文件一概不碰。所以**提交进仓库的代码永远是格式化过的**,你几乎无感。

`★ Insight ─────────────────────────────────────`
**为什么不直接在 hook 里写 `prettier --write` 而要套一层 lint-staged?** 因为"只格式化暂存文件"有几个棘手边界,lint-staged 帮你正确处理了:
1. **部分暂存**:你只 `git add` 了一个文件的前半部分(`git add -p`),lint-staged 会先把未暂存的改动 stash 起来,只格式化暂存的部分,再恢复——不会把你没打算提交的改动也卷进来。
2. **删除/重命名、文件名带空格** 等情况也都处理正确。
自己在 shell 里手写这些极易出错,等于重造 lint-staged。
`─────────────────────────────────────────────────`

补充:lint-staged 的配置可以放好几个地方(`package.json` 里的 `"lint-staged"` 字段、`.lintstagedrc`、`.lintstagedrc.json`、`.lintstagedrc.js`…)。我们选了**独立的 `.lintstagedrc.json`**,好处是不把 `package.json` 撑乱、一眼能看出职责。

（你之前看到的那次 hook 端到端验证——乱格式的 `const   x={a:1,b:2}` 被自动变成 `const x = { a: 1, b: 2 };`——走的就是这条链路。）
