AGENTS.md 可以理解成 **“给 Coding Agent 看的 README”**：README 给人看，AGENTS.md 给 Codex / Claude Code / Cursor Agent 这类工具看。它不要写成愿景文档，要写成**可执行的工程约束**。

OpenAI 官方 Codex 文档也建议可以放两层：全局 `~/.codex/AGENTS.md` 放个人长期偏好，仓库根目录 `AGENTS.md` 放项目规则；项目规则会用于约束该仓库里的 Codex 工作。([OpenAI开发者][1]) 另外 agents.md 社区也把它定义成一种面向 coding agent 的简单开放格式。([代理商.md][2])

## 一般结构

我建议你按这个顺序写：

```md
# AGENTS.md

## 1. Project overview

一句话说明这个仓库是干什么的。

## 2. Tech stack

列清楚前端、后端、包管理器、数据库、测试框架。

## 3. Common commands

安装、启动、构建、测试、lint、typecheck 命令。

## 4. Working rules

Agent 工作时必须遵守的流程，例如不能直接改 master，必须开 worktree，改代码前先读相关文件。

## 5. Code style

命名、目录、组件、API、错误处理、日志、类型约束。

## 6. Testing rules

什么时候必须补测试，怎么跑测试，最低验证标准是什么。

## 7. Git / PR rules

分支命名、commit 规则、PR 描述要求、禁止事项。

## 8. Safety / forbidden actions

不能删数据、不能改生产配置、不能引入大依赖、不能重写历史等。

## 9. Task completion checklist

Agent 每次结束前必须自检什么。
```

## 一个比较实用的模板

你可以直接从这个版本开始：

````md
# AGENTS.md

## Project overview

This repository is a Harness engineering infra project for standardizing AI coding agent workflows across multiple projects.

The goal is to provide reusable rules, skills, templates, and workflows for tools such as Codex, Claude Code, and Cursor.

## Tech stack

- Package manager: pnpm
- Runtime: Node.js
- Frontend: React / Next.js if applicable
- Backend: Node.js / API routes if applicable
- Database: specify if used
- Testing: specify actual test framework

## Common commands

Use these commands when working in this repository:

```bash
pnpm install
pnpm dev
pnpm lint
pnpm typecheck
pnpm test
pnpm build
```
````

If a command does not exist, check `package.json` before assuming the command name.

## Working rules

- Do not work directly on `master` or `main`.
- Before making changes, inspect the relevant files and understand the existing structure.
- Prefer small, focused changes over large rewrites.
- Do not refactor unrelated code while completing a task.
- Do not introduce new production dependencies without explicit approval.
- Preserve existing behavior unless the task explicitly asks to change it.
- When unsure, state the assumption clearly before proceeding.

## Worktree rules

- Use a separate git worktree for non-trivial feature work.
- Branch names should follow this pattern:

```bash
feature/<short-task-name>
fix/<short-bug-name>
chore/<short-maintenance-name>
docs/<short-doc-name>
```

- Never create feature work directly on `master` or `main`.
- Before starting, check current branch and git status.

## Code style

- Prefer clear, explicit code over clever abstractions.
- Keep functions small and cohesive.
- Use meaningful names.
- Avoid hidden global state.
- Avoid magic constants; name them when they matter.
- Public utilities should have comments or documentation when behavior is not obvious.
- Handle errors explicitly.
- Keep logs actionable and avoid noisy debug logs.

## Testing rules

- Run the relevant tests after changing code.
- Run `pnpm lint` and `pnpm typecheck` before considering the task complete.
- Add or update tests when:
  - fixing a bug
  - changing business logic
  - adding a new feature
  - changing public APIs

- If tests cannot be run, explain why and list what was checked manually.

## Git rules

- Keep commits focused.
- Do not commit generated files unless the project expects them.
- Do not modify lockfiles unless dependencies changed.
- Do not rewrite git history unless explicitly requested.
- Do not delete user work or untracked files.

## Documentation rules

- Update documentation when behavior, commands, setup, or workflow changes.
- Do not duplicate long requirements in this file.
- Link to dedicated docs when the details are too large for AGENTS.md.

## Forbidden actions

Do not do these unless explicitly requested:

- Delete files unrelated to the task.
- Change production environment configuration.
- Add large frameworks or dependencies.
- Disable tests, lint rules, or type checks to make the build pass.
- Replace existing architecture without discussion.
- Touch secrets, credentials, tokens, or `.env` values.
- Run destructive database or filesystem commands.

## Task completion checklist

Before finishing, verify:

- The change matches the user request.
- The implementation is minimal and focused.
- Relevant tests, lint, and type checks were run or the reason is documented.
- No unrelated files were modified.
- No secrets or local-only paths were introduced.
- The final response summarizes:
  - what changed
  - how it was verified
  - any remaining risks or follow-up work

````

## 你做 Harness 的话，重点要加这几类规则

普通项目的 AGENTS.md 只写“怎么跑项目”。但你这个 Harness 更重要的是**制度化 agent 行为**，所以建议额外加：

```md
## Harness principles

- This repo defines reusable AI coding agent workflows.
- Prefer rules that can be reused across projects.
- Separate universal rules from project-specific rules.
- Do not hardcode assumptions that only apply to one repository.
- When adding a new rule, include:
  - the problem it prevents
  - the expected behavior
  - an example if useful
````

再加一个：

```md
## Agent instruction design rules

- Instructions should be concrete and testable.
- Avoid vague rules like "write good code".
- Prefer rules like "run pnpm typecheck after changing TypeScript files".
- Keep instructions short enough for agents to read every session.
- Move long explanations into separate docs and link to them.
```

## 最佳实践：不要写太长

AGENTS.md 最怕变成“第二个知识库”。我的建议是：

```txt
AGENTS.md = 高频、强约束、每次都要看的规则
docs/*.md = 低频、长解释、背景知识
skills/* = 可复用的专项流程
scripts/* = 可以自动化执行的检查
```

比如：

```md
## References

- Engineering workflow: `docs/engineering-workflow.md`
- Worktree guide: `docs/git-worktree.md`
- Codex usage guide: `docs/codex-best-practices.md`
- Skill authoring guide: `docs/skills.md`
```

## 一个判断标准

写 AGENTS.md 时，每一条都问自己一句：

> 这条规则能不能改变 agent 的实际行为？

如果不能，就删掉。

比如：

```md
- Write high-quality code.
```

这句没用。

改成：

```md
- Prefer small functions with a single responsibility.
- Do not introduce shared abstractions until there are at least two real call sites.
- After changing TypeScript code, run `pnpm typecheck`.
```

这就有用了。

## 推荐你的落地方式

你可以先做三层：

```txt
~/.codex/AGENTS.md
```

放你的个人习惯，比如：

```md
- Prefer Chinese explanations when discussing architecture.
- Prefer pnpm.
- Ask before adding production dependencies.
- Always summarize verification steps.
```

项目根目录：

```txt
AGENTS.md
```

放这个仓库的硬规则，比如技术栈、命令、测试、分支、worktree。

子目录：

```txt
frontend/AGENTS.md
backend/AGENTS.md
infra/AGENTS.md
```

放局部规则，比如前端组件规范、后端 API 规范、infra 脚本规范。

一句话总结：

> AGENTS.md 不要写“项目介绍文”，要写“Agent 操作手册”；不要追求完整，追求每条规则都能约束一次真实编码行为。
