# LLM API Usage 字段与计费关系

> 理解 `usage` 中各 token 字段的关系，以及如何据此计算 API 费用。

## usage 字段中 token 数量的关系

| 字段 | 含义 | 示例值 |
|---|---|---|
| `input_tokens` | 总输入 token 数 | 5000 |
| `cache_read_input_tokens` | 其中命中缓存的 token 数 | 3000 |
| `cache_creation_input_tokens` | 本次新写入缓存的 token 数 | 0 |
| `output_tokens` | 模型生成的输出 token 数 | 1500 |

- **未命中缓存的 token 数** = `input_tokens - cache_read_input_tokens`（即 2000）
- `output_tokens` 与输入侧缓存无关，独立计费。
- `cache_creation_input_tokens` 为 0 表示本次没有新建缓存。

**核心关系**：

```
总输入 = 缓存命中部分 + 未缓存部分
总输出 = 独立计费（不受缓存影响）
```

---

## 价格计费示例（以 Claude Code 为例）

假设定价：

| 类别 | 单价（每百万 token） | 相对比例 |
|---|---|---|
| 输出 | $75 | — |
| 输入（基础） | $15 | 基准 |
| 缓存命中 | $1.5 | 基础价格的 10% |

基于上方示例 token 数量：

- **输出费用**：`1500 ÷ 1,000,000 × 75 = $0.1125`
- **未缓存输入费用**：`2000 ÷ 1,000,000 × 15 = $0.03`
- **缓存命中费用**：`3000 ÷ 1,000,000 × 1.5 = $0.0045`

**单次调用总成本** = $0.1125 + $0.03 + $0.0045 = **$0.147**

---

## 结论

API 账单由三部分分别累加计算，单价各不相同：

1. **输出 token** — 最贵
2. **未命中缓存的输入 token** — 中间价（基础价）
3. **命中缓存的输入 token** — 最便宜（通常为基础价的 10%）
