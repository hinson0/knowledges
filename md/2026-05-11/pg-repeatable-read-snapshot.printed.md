# Repeatable Read 快照一致性

> 来源:`fastapi_web/src/learning_demo/1900.md`(2026-05-11 流水笔记)

## 一句话总结

**Repeatable Read 给整个事务发一张"门票",票上印着事务开始那一刻的世界状态。事务期间不管外面怎么变,你看到的永远是票上那一刻的样子;事务结束、出门重新买票,才能看到新世界。**

## 三个关键点

### 1. 快照在哪一刻拍的

PostgreSQL 在事务里**第一条 SQL 语句执行时**拍快照(不是 `BEGIN` 那一刻),之后整个事务都用这一张。

### 2. RC vs RR 的本质区别

|                 | 快照粒度                  | 重复读结果 |
| --------------- | ------------------------- | ---------- |
| Read Committed  | **每条语句**拍一张新快照  | 可能变化   |
| Repeatable Read | **整个事务**共用一张快照  | 永远一致   |

### 3. "结束后能看到新值"很关键

实验里特别能说明问题:

- `COMMIT` 之前 → 看到 2000(老快照)
- `COMMIT` 之后再 `SELECT` → 看到 5000(新世界)

这证明数据**真的**已经被 B 改成 5000 了,A 只是被快照"隔离"了,看不到而已。

## 生活化比喻

想象你在看一份**报纸的早报**:

- **RC** = 你每读一篇文章前都要去拿最新一版,可能这篇是早报、下一篇就成了午报、晚报
- **RR** = 你拿一份早报回家慢慢看,外面就算出号外、出晚报,你手上这份始终是早上那版

## 什么时候该用 RR

**Why:** 需要**事务内多次读取保持一致**的场景:

- 生成报表(多张表的数据要对得上)
- 对账、统计
- "先查再改"的业务逻辑(避免基于过期数据做决策)

**How to apply:** 用 `BEGIN ISOLATION LEVEL REPEATABLE READ;` 开启。代价是**并发冲突时可能报错**(`could not serialize access due to concurrent update`),需要应用层捕获并重试。

简单记:**RC 看实时,RR 看快照**。
